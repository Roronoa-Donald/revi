Chapitre 4

Status codes

Dire clairement ce qui s est passe.

Codes frequents

Le code HTTP fait partie de la reponse. Il explique le resultat technique de la requete.

Une API claire utilise les bons codes au lieu de tout renvoyer en 200.

CodeSens
200OK
201Created
400Bad Request
401Non authentifie
403Interdit
404Ressource introuvable
500Erreur serveur
Memo rapide : 401 = qui es-tu ? 403 = je te connais mais tu n as pas le droit.

Erreurs JSON

Meme en erreur, une API peut renvoyer un JSON comprehensible.

Le frontend peut alors afficher le message proprement.

return jsonify({"error": "student_not_found"}), 404
Tip : Une erreur utile a un code HTTP et un message machine lisible.

Validation

Si un champ obligatoire manque, on renvoie 400.

Si une ressource n existe pas, on renvoie 404.

Piege : Ne renvoie pas 500 pour une erreur utilisateur. 500 signifie probleme serveur.

Questions de fin de chapitre

Reponds avant de lire la correction. Elle se debloque apres une bonne reponse, ou apres deux erreurs.

1. Quelle signification générale ont les codes HTTP de la famille 2xx ?
Correction : La famille 2xx (ex: 200, 201, 204) représente les réussites d'opérations.
2. Quel est l'usage standard du code de statut '200 OK' ?
Correction : Le code 200 indique que la requête s'est déroulée avec succès et que le résultat attendu est renvoyé dans le corps de réponse.
3. Quel code HTTP doit idéalement renvoyer une route POST suite à la création réussie d'une ressource ?
Correction : 201 Created est le code standard pour notifier le client qu'une nouvelle ressource a été correctement insérée et possède désormais son propre URI.
4. Quelle signification a le code '400 Bad Request' ?
Correction : Le code 400 indique une erreur de la part du client (ex: format JSON invalide, champs obligatoires manquants).
5. Que signifie précisément le code HTTP '401 Unauthorized' ?
Correction : Le code 401 signifie que la requête requiert une authentification utilisateur qui a échoué ou est absente.
