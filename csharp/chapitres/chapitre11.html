Module 11

Gestion d'Erreurs Try / Catch

Sharpy a un problème : son programme plante quand l'utilisateur tape « abc » au lieu d'un nombre. Un programme professionnel ne plante jamais sans prévenir. Les blocs try/catch sont le filet de sécurité du développeur.

Pourquoi gérer les erreurs ?

En C#, quand quelque chose tourne mal (division par zéro, fichier introuvable, entrée invalide), le runtime lance une exception — un objet qui décrit le problème. Si personne ne l'attrape, le programme crash.

C'est quoi une exception ?

Une exception est un objet qui hérite de System.Exception. Elle contient un message (.Message), une pile d'appels (.StackTrace) et parfois une exception interne (.InnerException).

Les exceptions les plus courantes que tu rencontreras :

Exception Cause
FormatExceptionConversion de texte invalide
DivideByZeroExceptionDivision par zéro
IndexOutOfRangeExceptionIndex hors limites d'un tableau
NullReferenceExceptionAccès à un objet null
FileNotFoundExceptionFichier introuvable
ArgumentExceptionArgument invalide passé à une méthode
Le cauchemar du développeur

La NullReferenceException est l'erreur la plus fréquente en C#. Elle survient quand tu essaies d'utiliser un objet qui vaut null. Tony Hoare, l'inventeur du null, l'a appelée son « erreur à un milliard de dollars ».

Try / Catch : le filet de sécurité

La structure est simple : tu essaies (try) d'exécuter du code, et si une exception est lancée, tu l'attrapes (catch) pour la gérer proprement.

Syntaxe try/catch
Console.Write("Entre un nombre : ");
string saisie = Console.ReadLine();

try
{
    int nombre = int.Parse(saisie);
    Console.WriteLine($"Le double est : {nombre * 2}");
}
catch (FormatException ex)
{
    Console.WriteLine($"Erreur : '{saisie}' n'est pas un nombre valide.");
    Console.WriteLine($"Détail : {ex.Message}");
}

Si l'utilisateur tape abc, au lieu de crasher, le programme affiche un message d'erreur lisible et continue.

Attraper plusieurs types d'exceptions

Tu peux chaîner plusieurs blocs catch pour gérer différents types d'erreurs. Place toujours les exceptions les plus spécifiques en premier, car C# prend le premier catch qui correspond.

Catch multiples
try
{
    int[] tab = { 1, 2, 3 };
    Console.WriteLine(tab[10]);  // Index hors limites !
}
catch (IndexOutOfRangeException ex)
{
    Console.WriteLine("Index invalide !");
}
catch (Exception ex)  // Attrape tout le reste
{
    Console.WriteLine($"Erreur inattendue : {ex.Message}");
}
Ne jamais « avaler » une exception

Un catch vide (catch { }) est interdit en code professionnel. Il masque le problème : ton programme ne crash plus, mais produit des résultats incorrects en silence. Toujours logger ou afficher le message.

Finally : le nettoyage garanti

Le bloc finally s'exécute toujours, qu'une exception ait été lancée ou non. C'est l'endroit idéal pour fermer des fichiers, des connexions réseau ou libérer des ressources.

Try / Catch / Finally
StreamReader lecteur = null;
try
{
    lecteur = new StreamReader("data.txt");
    string contenu = lecteur.ReadToEnd();
    Console.WriteLine(contenu);
}
catch (FileNotFoundException)
{
    Console.WriteLine("Fichier introuvable !");
}
finally
{
    // Toujours exécuté, même s'il y a eu une erreur
    lecteur?.Close();
    Console.WriteLine("Nettoyage effectué.");
}
Le mot-clé using : finally automatique

En C#, le mot-clé using crée automatiquement un bloc try/finally qui appelle .Dispose() à la fin. C'est la méthode recommandée pour gérer les fichiers, les connexions, etc.

using = try/finally automatique
// Équivalent au try/finally ci-dessus, mais plus propre
using (StreamReader lecteur = new StreamReader("data.txt"))
{
    string contenu = lecteur.ReadToEnd();
    Console.WriteLine(contenu);
}
// lecteur.Dispose() est appelé automatiquement ici

// Syntaxe C# 8+ encore plus concise :
using var reader = new StreamReader("data.txt");
string texte = reader.ReadToEnd();
// Dispose automatique en fin de portée

Throw : lancer ses propres exceptions

Tu n'es pas obligé d'attendre qu'une erreur survienne. Tu peux lancer une exception toi-même quand une condition métier n'est pas respectée.

throw
static void Retirer(double solde, double montant)
{
    if (montant <= 0)
        throw new ArgumentException("Le montant doit être positif.");

    if (montant > solde)
        throw new InvalidOperationException("Solde insuffisant !");

    Console.WriteLine($"Retrait de {montant}€ effectué.");
}
throw vs throw ex

Quand tu relances une exception dans un catch, utilise throw; (sans argument) pour conserver la pile d'appels originale. throw ex; réinitialise le StackTrace et tu perds l'information de débogage.

Tu peux aussi créer tes propres exceptions en héritant de Exception :

Exception personnalisée
public class SoldeInsuffisantException : Exception
{
    public double Solde { get; }
    public double Montant { get; }

    public SoldeInsuffisantException(double solde, double montant)
        : base($"Solde ({solde}€) insuffisant pour retirer {montant}€")
    {
        Solde = solde;
        Montant = montant;
    }
}

// Utilisation :
throw new SoldeInsuffisantException(100, 250);
Bonnes pratiques résumées
  • Utilise try/catch uniquement quand tu peux gérer l'erreur
  • Préfère TryParse à Parse + catch pour les conversions
  • Utilise using pour les ressources (fichiers, connexions)
  • N'attrape jamais Exception sans raison — sois spécifique
  • Utilise throw; et non throw ex; pour relancer
