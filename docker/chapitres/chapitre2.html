Chapitre 2

Docker, images et registres

Engine, client, Docker Hub, pull, run et layers

Architecture Docker

Docker fonctionne comme une application client-serveur. Le client en ligne de commande reçoit les ordres, puis le daemon du Docker Engine gère réellement les images, conteneurs, réseaux et volumes.

Élément Rôle
Client Docker Interface CLI: docker run, docker ps, docker build...
Daemon Processus persistant qui gère les conteneurs.
Docker Engine Environnement d'exécution et services Docker.
Registre Serveur qui stocke les images.

Registres et Docker Hub

Un registre Docker permet de récupérer des images avec pull et d'en publier avec push. Docker Hub est le registre public le plus connu.

  • Image officielle: par exemple ubuntu, nginx ou redis.
  • Image utilisateur: par exemple compte/image.
  • Image privée ou self-hosted: registry.example.com:5000/projet/image.
  • Un tag comme latest ou v1.0 sert de référence vers une image.

Commandes de base

Les premières commandes à maîtriser sont docker pull et docker run.

$ docker pull redis
$ docker run hello-world
$ docker run -it ubuntu
$ docker ps

Images en couches

Une image Docker est composée de layers en lecture seule. Lorsqu'un conteneur s'exécute, Docker ajoute une couche writable propre au conteneur.

Grâce au Copy-on-Write, les couches communes sont partagées et seules les modifications sont écrites dans la couche supérieure.

  • Les layers économisent de l'espace disque.
  • Ils accélèrent les téléchargements si certaines couches existent déjà.
  • Ils expliquent pourquoi un conteneur supprimé perd sa couche writable.
  • Ils sont liés aux instructions du Dockerfile.
