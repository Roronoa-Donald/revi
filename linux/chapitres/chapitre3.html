Module 03

Droits & Permissions

Sami découvre pourquoi il ne peut pas tout faire — et c'est tant mieux. La sécurité fichier sous Linux.

Comprendre les permissions

Quand Sami tape ls -l, il voit quelque chose comme ça :

bash-rwxr-xr-- 1 sami devops 4096 jan 15 10:00 script.sh
drwxr-x--- 2 sami devops 4096 jan 15 09:00 projets/

Décomposons le premier caractère et les 9 suivants :

Décodage des permissions

-rwxr-xr-- se lit en 4 parties :
1. Type : - = fichier, d = dossier, l = lien symbolique
2. Propriétaire (user) : rwx = lecture + écriture + exécution
3. Groupe : r-x = lecture + exécution (pas d'écriture)
4. Autres (other) : r-- = lecture seule

Chaque permission a une valeur octale :

r (read)    = 4
w (write)   = 2
x (execute) = 1
- (rien)    = 0

Exemples :
rwx = 4+2+1 = 7
r-x = 4+0+1 = 5
r-- = 4+0+0 = 4
--- = 0+0+0 = 0

Donc rwxr-xr-- = 754

chmod — Modifier les permissions

Notation octale (numérique)

bashchmod 755 script.sh       # rwxr-xr-x (commun pour les scripts)
chmod 644 fichier.txt     # rw-r--r-- (commun pour les fichiers)
chmod 700 prive/          # rwx------ (toi seul y accèdes)
chmod 600 secret.key      # rw------- (fichier sensible)
chmod -R 755 dossier/   # Récursif : applique à tout le dossier

Notation symbolique

bashchmod u+x script.sh      # Ajoute exécution pour le propriétaire
chmod g-w fichier.txt    # Retire écriture pour le groupe
chmod o=r doc.txt        # Fixe "autres" à lecture seule
chmod a+r public.html    # Ajoute lecture pour tout le monde (a=all)
chmod u+x,g+r fichier   # Plusieurs changements d'un coup
u, g, o, a

u = user (propriétaire), g = group, o = other, a = all. Utilise + pour ajouter, - pour retirer, = pour fixer exactement.

Piège — exécution sur les dossiers

Pour un fichier, x = exécutable. Pour un dossier, x = le droit d'y entrer (cd). Sans x sur un dossier, même avec r, tu ne peux pas lister son contenu !

chown et chgrp — Changer le propriétaire

bash# Changer le propriétaire
sudo chown sami fichier.txt

# Changer propriétaire ET groupe
sudo chown sami:devops fichier.txt

# Récursif sur un dossier
sudo chown -R www-data:www-data /var/www/html/

# Changer uniquement le groupe
sudo chgrp devops projet/
Pourquoi sudo ?

Seul root peut changer le propriétaire d'un fichier. C'est logique : sinon n'importe qui pourrait s'attribuer les fichiers des autres ! Par contre, tu peux changer le groupe d'un fichier qui t'appartient si tu fais partie du groupe cible.

umask — Les permissions par défaut

Quand tu crées un fichier, il n'a pas les permissions 777. Le umask définit un masque qui retire des permissions :

bashumask                     # Affiche le masque actuel (souvent 0022)
0022

# Calcul : permissions finales = permissions max - umask
# Fichier : 666 - 022 = 644 (rw-r--r--)
# Dossier : 777 - 022 = 755 (rwxr-xr-x)

umask 0077                # Change : fichiers = 600, dossiers = 700
Pourquoi 666 et pas 777 pour les fichiers ?

Par sécurité, les fichiers sont créés sans le bit x (exécution). Sinon tous les fichiers seraient exécutables par défaut — imaginer un fichier texte qui s'exécute comme un programme !

Permissions spéciales (SUID, SGID, Sticky)

bash# SUID (Set User ID) — s'exécute avec les droits du propriétaire
chmod 4755 programme      # ou chmod u+s programme
# Exemple : /usr/bin/passwd a le SUID pour modifier /etc/shadow

# SGID (Set Group ID) — les fichiers créés héritent du groupe du dossier
chmod 2755 dossier/      # ou chmod g+s dossier/
# Pratique pour les dossiers de projet partagés

# Sticky bit — seul le propriétaire peut supprimer ses fichiers
chmod 1777 /tmp          # ou chmod +t /tmp
# /tmp utilise le sticky bit : tout le monde écrit, personne ne supprime les fichiers des autres
Ne pas confondre

Le SUID sur un script est ignoré par Linux (sécurité). Il ne fonctionne que sur les binaires compilés. Ne mets JAMAIS le SUID sur un script shell !
