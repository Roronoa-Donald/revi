Module 11

Héritage & Polymorphisme

JavaDuke a créé ses premières classes. Mais il doit maintenant modéliser un zoo avec des dizaines d'animaux différents. Son mentor CaféBot lui dit : « Un Chien est un Animal. Un Chat est un Animal. L'héritage, c'est cette relation IS-A — construire des classes enfants à partir d'un parent commun. »

Le mot-clé extends — créer une hiérarchie

L'héritage c'est comme un arbre généalogique. Le parent transmet ses gènes (attributs + méthodes) à ses enfants. L'enfant hérite de tout, mais peut aussi développer ses propres caractéristiques. En Java, une classe extends une autre pour hériter de tout son contenu.

// Classe parent (superclasse)
public class Animal {
    protected String nom;
    protected int age;

    public Animal(String nom, int age) {
        this.nom = nom;
        this.age = age;
    }

    public void manger() {
        System.out.println(nom + " mange.");
    }

    public void parler() {
        System.out.println(nom + " fait un bruit.");
    }
}

// Classe enfant (sous-classe) — hérite de Animal
public class Chien extends Animal {
    private String race;

    public Chien(String nom, int age, String race) {
        super(nom, age);  // Appelle le constructeur du parent
        this.race = race;
    }

    // Méthode propre au Chien
    public void rapporter() {
        System.out.println(nom + " rapporte la balle !");
    }
}

// Utilisation
Chien rex = new Chien("Rex", 3, "Berger");
rex.manger();       // Hérité de Animal
rex.parler();       // Hérité de Animal
rex.rapporter();    // Propre à Chien
Sortie : Rex mange. Rex fait un bruit. Rex rapporte la balle !
💡 protected vs private : Un attribut private n'est PAS accessible par les enfants. Utilise protected pour que les sous-classes y accèdent directement, ou garde private avec des getters/setters.
⚠️ Java = héritage simple : Une classe ne peut hériter que d'une seule classe parent. Pas d'héritage multiple ! class C extends A, B → Erreur. Pour contourner : utilise les interfaces (voir plus bas).

super — communiquer avec le parent

Le mot-clé super permet d'accéder au constructeur et aux méthodes de la classe parent. C'est comme appeler ses parents quand on a besoin de leur aide.

public class Chat extends Animal {
    private boolean estInterieur;

    public Chat(String nom, int age, boolean estInterieur) {
        super(nom, age);  // OBLIGATOIRE en 1ère ligne !
        this.estInterieur = estInterieur;
    }

    // On peut aussi appeler super.methode()
    public void afficherInfos() {
        super.manger();  // Appelle manger() du parent
        System.out.println("Type : " + (estInterieur ? "d'intérieur" : "d'extérieur"));
    }
}
⚠️ Règle d'or : super() doit toujours être la première instruction du constructeur enfant. Si le parent n'a pas de constructeur sans paramètres, tu es obligé d'appeler super(params) explicitement — sinon erreur de compilation.

@Override — redéfinir une méthode parent

Ton parent t'a appris à cuisiner des pâtes. Mais toi, tu ajoutes ta propre sauce spéciale. C'est ça @Override : tu gardes le nom de la recette, mais tu changes la méthode. L'enfant hérite de la compétence, mais la personnalise.

public class Chien extends Animal {
    // ... constructeur ...

    @Override
    public void parler() {
        System.out.println(nom + " aboie : Woof ! 🐕");
    }
}

public class Chat extends Animal {
    // ... constructeur ...

    @Override
    public void parler() {
        System.out.println(nom + " miaule : Miaou ! 🐱");
    }
}

// Test
Animal a1 = new Chien("Rex", 3, "Berger");
Animal a2 = new Chat("Mimi", 2, true);
a1.parler();   // Appelle la version Chien
a2.parler();   // Appelle la version Chat
Sortie : Rex aboie : Woof ! 🐕 Mimi miaule : Miaou ! 🐱
💡 Pourquoi @Override ? L'annotation n'est pas obligatoire mais fortement recommandée. Elle permet au compilateur de vérifier que tu redéfinis bien une méthode existante du parent. Si tu fais une faute de frappe dans le nom, le compilateur te prévient au lieu de créer une nouvelle méthode.

Polymorphisme — un même appel, des comportements différents

Imagine un bouton « Play » sur une télécommande. Tu appuies dessus devant une TV → elle diffuse une émission. Devant un lecteur CD → il joue de la musique. Même bouton, même action, mais résultat différent selon l'appareil. C'est ça le polymorphisme.

// Un tableau d'Animal peut contenir des Chien, Chat, etc.
Animal[] zoo = new Animal[4];
zoo[0] = new Chien("Rex", 3, "Berger");
zoo[1] = new Chat("Mimi", 2, true);
zoo[2] = new Chien("Buddy", 5, "Labrador");
zoo[3] = new Chat("Félix", 1, false);

// Polymorphisme : même méthode, comportement différent
for (Animal animal : zoo) {
    animal.parler();  // Java choisit la bonne version automatiquement
}

// instanceof — vérifier le type réel
for (Animal animal : zoo) {
    if (animal instanceof Chien) {
        Chien c = (Chien) animal;  // Downcasting
        c.rapporter();
    }
}
Sortie : Rex aboie : Woof ! 🐕 Mimi miaule : Miaou ! 🐱 Buddy aboie : Woof ! 🐕 Félix miaule : Miaou ! 🐱 Rex rapporte la balle ! Buddy rapporte la balle !
💡 Upcasting vs Downcasting :
• Upcasting (implicite) : Animal a = new Chien(...) — toujours sûr.
• Downcasting (explicite) : Chien c = (Chien) animal — dangereux ! Utilise instanceof avant pour éviter un ClassCastException.

Classes abstraites — le contrat incomplet

Une classe abstraite c'est comme un plan architectural qui dit « il y aura une porte ici » sans préciser si c'est une porte en bois ou en métal. Elle définit le quoi, les enfants définissent le comment.

// Classe abstraite — ne peut PAS être instanciée
public abstract class Forme {
    protected String couleur;

    public Forme(String couleur) {
        this.couleur = couleur;
    }

    // Méthode abstraite — PAS de corps, les enfants DOIVENT l'implémenter
    public abstract double aire();

    // Méthode concrète — les enfants en héritent
    public void afficher() {
        System.out.printf("Forme %s, aire = %.2f%n", couleur, aire());
    }
}

public class Cercle extends Forme {
    private double rayon;

    public Cercle(String couleur, double rayon) {
        super(couleur);
        this.rayon = rayon;
    }

    @Override
    public double aire() {
        return Math.PI * rayon * rayon;
    }
}

public class Rectangle extends Forme {
    private double largeur, hauteur;

    public Rectangle(String couleur, double l, double h) {
        super(couleur);
        this.largeur = l;
        this.hauteur = h;
    }

    @Override
    public double aire() {
        return largeur * hauteur;
    }
}

// Utilisation polymorphe
Forme[] formes = { new Cercle("rouge", 5), new Rectangle("bleu", 4, 6) };
for (Forme f : formes) {
    f.afficher();
}
Sortie : Forme rouge, aire = 78.54 Forme bleu, aire = 24.00

Interfaces — le contrat à respecter

Une interface c'est comme un contrat de travail : « Tu dois savoir faire X, Y et Z ». Peu importe comment tu le fais, tant que tu respectes le contrat. Contrairement à l'héritage (qui est limité à 1 parent), une classe peut implémenter plusieurs interfaces.

// Interface = contrat pur
public interface Nageur {
    void nager();  // Pas de corps ! implicitly public abstract
}

public interface Volant {
    void voler();
}

// Une classe peut implémenter PLUSIEURS interfaces
public class Canard extends Animal implements Nageur, Volant {
    public Canard(String nom, int age) {
        super(nom, age);
    }

    @Override
    public void parler() {
        System.out.println(nom + " fait : Coin coin ! 🦆");
    }

    @Override
    public void nager() {
        System.out.println(nom + " nage dans l'étang.");
    }

    @Override
    public void voler() {
        System.out.println(nom + " s'envole !");
    }
}

Canard donald = new Canard("Donald", 5);
donald.parler();
donald.nager();
donald.voler();
Sortie : Donald fait : Coin coin ! 🦆 Donald nage dans l'étang. Donald s'envole !

🎉 Résumé :
• extends → hérite d'une classe (1 seul parent)
• super() → appelle le constructeur parent
• @Override → redéfinit une méthode héritée
• abstract → classe non-instanciable avec méthodes sans corps
• interface → contrat pure, implements multiple
• instanceof → vérifie le type réel d'un objet
• Polymorphisme → même appel, comportement adapté au type réel
🧠 Mémo — « extends = EST UN, implements = SAIT FAIRE »
• class Chien extends Animal → un chien EST UN animal
• class Chien implements Nageur → un chien SAIT nager
• super() → appeler le constructeur du parent
Règle : 1 seul extends, mais plusieurs implements.
⚠ Classe abstraite : impossible de faire new ClasseAbstraite(). Il faut une classe fille concrète.
@Override = « je remplace la méthode du parent ». Le compilateur vérifie pour toi.
