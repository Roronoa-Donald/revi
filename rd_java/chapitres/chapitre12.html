Module 12

Exceptions : try / catch / finally

JavaDuke a écrit un super programme... mais il plante dès qu'un utilisateur tape « pizza » au lieu d'un nombre. CaféBot lui dit : « Un bon programme ne plante jamais — il gère les erreurs. C'est le rôle des exceptions. »

Quand ça plante — comprendre les exceptions

Une exception c'est comme un accident sur la route. Sans gestion, le trafic est bloqué (le programme crashe). Avec un plan de déviation (try/catch), tu contournes le problème et le trafic continue.

// Sans gestion d'erreur — le programme CRASHE
int[] tab = {1, 2, 3};
System.out.println(tab[10]);  // 💥 ArrayIndexOutOfBoundsException

String texte = null;
System.out.println(texte.length());  // 💥 NullPointerException

int n = Integer.parseInt("pizza");  // 💥 NumberFormatException

int r = 10 / 0;  // 💥 ArithmeticException
💥 Résultat sans try/catch : Exception in thread "main" java.lang.ArrayIndexOutOfBoundsException: Index 10 out of bounds for length 3 at Main.main(Main.java:3) ⚠️ Le programme s'arrête immédiatement. Rien après cette ligne ne s'exécute.

try / catch — attraper les erreurs

Le bloc try contient le code risqué. Le bloc catch contient le plan B si une erreur survient. Le programme ne crashe plus — il gère l'erreur et continue.

import java.util.Scanner;

Scanner sc = new Scanner(System.in);

try {
    System.out.print("Entre un nombre : ");
    int n = Integer.parseInt(sc.nextLine());
    int resultat = 100 / n;
    System.out.println("100 / " + n + " = " + resultat);
} catch (NumberFormatException e) {
    System.out.println("❌ Ce n'est pas un nombre valide !");
} catch (ArithmeticException e) {
    System.out.println("❌ Division par zéro impossible !");
}

System.out.println("✅ Le programme continue normalement.");
Sortie (si l'utilisateur tape "pizza") : Entre un nombre : pizza ❌ Ce n'est pas un nombre valide ! ✅ Le programme continue normalement.
Sortie (si l'utilisateur tape "0") : Entre un nombre : 0 ❌ Division par zéro impossible ! ✅ Le programme continue normalement.
💡 Ordre des catch : Mets les exceptions les plus spécifiques en premier. Si tu mets catch (Exception e) en premier, il attrape tout et les catch suivants deviennent inaccessibles (erreur de compilation). 
// Catch multiple (Java 7+) — un seul catch pour plusieurs types
try {
    // code risqué...
} catch (NumberFormatException | ArithmeticException e) {
    System.out.println("Erreur : " + e.getMessage());
}

// Catch générique — attrape TOUTES les exceptions
try {
    // code risqué...
} catch (Exception e) {
    System.out.println("Erreur inattendue : " + e.getMessage());
    e.printStackTrace();  // Affiche la trace complète (pour debug)
}
⚠️ MAUVAISE PRATIQUE : Un catch vide catch (Exception e) { } est le pire anti-pattern en Java. Tu masques l'erreur silencieusement, le bug devient invisible et indétectable. Au minimum, log le message d'erreur.

finally — le code qui s'exécute TOUJOURS

Pense au finally comme l'acte de « ranger sa chambre en partant ». Que tu aies réussi ton code (try) ou qu'une erreur ait eu lieu (catch), le finally s'exécute dans tous les cas. Parfait pour fermer des ressources (fichiers, connexions, Scanner).

Scanner sc = null;
try {
    sc = new Scanner(System.in);
    System.out.print("Ton âge : ");
    int age = Integer.parseInt(sc.nextLine());
    System.out.println("Tu as " + age + " ans.");
} catch (NumberFormatException e) {
    System.out.println("❌ Âge invalide !");
} finally {
    System.out.println("🔒 Nettoyage effectué (finally toujours exécuté)");
    if (sc != null) sc.close();
}
Sortie (saisie invalide) : Ton âge : abc ❌ Âge invalide ! 🔒 Nettoyage effectué (finally toujours exécuté)
💡 try-with-resources (Java 7+) : Pour les ressources qui implémentent AutoCloseable (Scanner, BufferedReader, FileWriter…), utilise cette syntaxe plus propre — le close() est automatique : 
// try-with-resources — close() automatique !
try (Scanner sc = new Scanner(System.in)) {
    System.out.print("Nombre : ");
    int n = sc.nextInt();
    System.out.println("Double : " + n * 2);
} catch (Exception e) {
    System.out.println("Erreur : " + e.getMessage());
}
// sc est automatiquement fermé à la fin du try, même en cas d'exception

throw & throws — lancer et déclarer des exceptions

throw lance une exception manuellement. throws dans la signature d'une méthode déclare qu'elle peut lancer une exception — l'appelant devra la gérer.

// throw — lancer manuellement une exception
public static void setAge(int age) {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("Âge invalide : " + age);
    }
    System.out.println("Âge défini à " + age);
}

// Appel
try {
    setAge(-5);
} catch (IllegalArgumentException e) {
    System.out.println("Erreur : " + e.getMessage());
}
Sortie : Erreur : Âge invalide : -5
// throws — déclarer les exceptions possibles dans la signature
public static int lireFichier(String chemin) throws IOException {
    BufferedReader reader = new BufferedReader(new FileReader(chemin));
    String ligne = reader.readLine();
    reader.close();
    return Integer.parseInt(ligne);
}

// L'appelant DOIT gérer l'exception
try {
    int val = lireFichier("data.txt");
} catch (IOException e) {
    System.out.println("Fichier introuvable !");
}

Checked vs Unchecked — les deux familles

Les exceptions checked sont comme des accidents prévisibles — le compilateur t'oblige à prévoir un plan. Les unchecked sont des bugs de programmation — des erreurs que tu aurais dû éviter dans ton code.

Checked (vérifiées)Unchecked (non vérifiées)
Héritage↳ Exception↳ RuntimeException

ExemplesIOException, SQLException, FileNotFoundExceptionNullPointerException, NumberFormatException, ArrayIndexOutOfBoundsException

💡 Hiérarchie des exceptions :
Throwable → Error (ne pas attraper) + Exception
Exception → Checked (IOException, etc.) + RuntimeException → Unchecked (NPE, etc.)

Exceptions personnalisées & bonnes pratiques

// Exception personnalisée
public class SoldeInsuffisantException extends Exception {
    private double montant;

    public SoldeInsuffisantException(double montant) {
        super("Solde insuffisant pour retirer " + montant + " €");
        this.montant = montant;
    }

    public double getMontant() { return montant; }
}

// Utilisation dans une classe CompteBancaire
public void retirer(double montant) throws SoldeInsuffisantException {
    if (montant > solde) {
        throw new SoldeInsuffisantException(montant);
    }
    solde -= montant;
}

// Appel
try {
    compte.retirer(5000);
} catch (SoldeInsuffisantException e) {
    System.out.println(e.getMessage());
}
Sortie : Solde insuffisant pour retirer 5000.0 €
✅ Bonnes pratiques — la checklist de CaféBot :
1. Ne catch JAMAIS un catch (Exception e) { } vide
2. Catch spécifique d'abord, générique ensuite
3. Utilise try-with-resources pour les fichiers/Scanner
4. Préfère corriger plutôt que catch les unchecked
5. Crée tes propres exceptions pour des erreurs métier
6. finally ou try-with-resources pour libérer les ressources
7. Log ou affiche e.getMessage(), jamais un catch silencieux
