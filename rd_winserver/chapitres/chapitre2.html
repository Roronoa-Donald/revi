Haute Disponibilité & Clustering

HA, redondance, failover, monitoring, SLA et les 5 types de clustering.

Qu'est-ce que la Haute Disponibilité (HA) ?

La haute disponibilité (souvent abrégée HA, de l'anglais High Availability) est un ensemble de mécanismes et de configurations techniques qui permettent à un système informatique de continuer à fonctionner même en cas de panne d'un de ses composants (serveur, disque, réseau, etc.).

Objectif : Minimiser ou éviter les interruptions de service pour garantir un fonctionnement continu, surtout pour les applications critiques (banques, e-commerce, hôpitaux, etc.).

Les 3 Piliers de la HA

La haute disponibilité repose sur trois mécanismes :

1. Redondance

La redondance consiste à dupliquer des composants critiques d'un système (serveurs, disques, alimentations, connexions réseau, etc.) afin de garantir la continuité de service en cas de défaillance. L'objectif est d'éviter un point unique de défaillance : si un élément tombe en panne, un autre, déjà prêt et fonctionnel, peut prendre le relais.

Exemple : Deux serveurs web configurés en parallèle : si l'un tombe, l'autre continue à répondre aux requêtes.

2. Basculement automatique (Failover)

Le failover est un mécanisme permettant de transférer automatiquement l'activité d'un composant défaillant vers un autre composant redondant, sans intervention humaine et avec un minimum d'interruption de service.

Exemple : Si une base de données principale devient inaccessible, le système bascule automatiquement sur une base de secours.

3. Supervision (Monitoring)

La supervision désigne l'ensemble des outils et processus permettant de surveiller en temps réel l'état de santé des ressources informatiques (serveurs, services, performances, etc.). Elle permet de détecter les anomalies et, dans un système redondant, de déclencher automatiquement un basculement.

Exemple : Un outil comme Zabbix ou Nagios vérifie constamment si les serveurs sont disponibles et alerte (ou agit) dès qu'une panne est détectée.

Niveaux de Disponibilité (SLA)

La disponibilité s'exprime souvent en pourcentage par an :

DisponibilitéIndisponibilité / an
99 %3 jours 15 h 36 min
99,9 %8 h 45 min 36 s
99,99 %52 min 34 s
99,999 %≈ 5 min 15 s
99,9999 %≈ 31,5 secondes

Exemple Concret : Load Balancer

Un site web est hébergé sur 2 serveurs web avec un équilibreur de charge (load balancer) devant eux. Si un serveur tombe :

  1. Le load balancer détecte l'échec.
  2. Il redirige les utilisateurs vers le serveur qui fonctionne.
  3. L'utilisateur ne remarque aucune interruption.

HA en Environnement Virtualisé

Dans un environnement comme Proxmox ou VMware vSphere :

  • On peut créer un cluster de plusieurs hôtes.
  • Les machines virtuelles peuvent être automatiquement redémarrées sur un autre hôte en cas de panne matérielle.
  • Si un hyperviseur tombe, ses VM redémarrent ailleurs (selon les règles HA configurées).

Le Clustering

Le clustering (ou grappes de serveurs) est une technique d'architecture informatique qui consiste à regrouper plusieurs serveurs (ou nœuds) pour qu'ils fonctionnent ensemble comme un seul système afin de garantir :

  • La haute disponibilité
  • La répartition de charge
  • La tolérance aux pannes
Exemple : Une application web importante (site d'e-commerce) est déployée sur 3 serveurs en cluster. Si l'un des serveurs tombe en panne, les autres prennent automatiquement le relais (failover). Le trafic est partagé entre les serveurs (load balancing). Cela évite les interruptions de service.

Les 5 Types de Clustering

1. Cluster Active-Passive

Un ou plusieurs nœuds sont actifs et traitent les requêtes. Les autres sont passifs (en veille), prêts à prendre le relais si un nœud actif tombe.

Exemple : Serveur A est actif. Serveur B est en veille. Si A tombe, B devient actif automatiquement.

AvantagesInconvénientsCas d'usage
Configuration simple
Moins de conflits de synchronisation		 Serveur passif ne travaille pas (sous-utilisation)
Temps de basculement peut être perceptible		 Applications critiques mais à faible charge constante
Services financiers, ERP

2. Cluster Active-Active

Tous les nœuds sont actifs en même temps. Ils partagent les tâches de traitement. S'ils sont bien configurés, ils se remplacent automatiquement en cas de panne.

Exemple : 3 serveurs d'une application web partagent les requêtes des utilisateurs.

AvantagesInconvénientsCas d'usage
Meilleure performance (répartition)
Haute tolérance aux pannes
Aucun nœud inactif		 Configuration complexe
Gestion des conflits/synchronisation plus difficile		 Sites web à fort trafic
Infrastructures virtualisées

3. Load Balancing Cluster

Un répartiteur de charge (load balancer) distribue les requêtes entre plusieurs serveurs actifs. Utilisé principalement pour améliorer la performance et la scalabilité.

Exemple : Un site web reçoit 1000 requêtes/seconde, réparties sur 4 serveurs web via un load balancer.

AvantagesInconvénientsCas d'usage
Équilibrage du trafic
Mise à l'échelle horizontale facile
Améliore disponibilité et réactivité		 Ne remplace pas un failover complet
Load balancer = point critique (sauf s'il est redondant)		 Serveurs web, serveurs d'applications, frontaux API

4. Failover Cluster (Cluster de basculement)

Chaque nœud partage un même espace de stockage. Si un nœud échoue, un autre reprend les services avec le même état, via le stockage partagé.

Exemple : Deux nœuds SQL Server partagent un disque SAN. En cas de panne, le deuxième nœud démarre le service SQL avec les mêmes données.

AvantagesInconvénientsCas d'usage
Tolérance élevée aux pannes
Restauration automatique des services
Haute cohérence des données		 Besoin d'un stockage partagé
Dépend de la qualité du SAN ou NAS		 Bases de données critiques
Applications métier vitales

5. Grid Cluster / Compute Cluster

Utilisé pour le calcul intensif distribué. Plusieurs nœuds traitent en parallèle des calculs complexes.

Exemple : Simulation météo, rendu 3D, intelligence artificielle.

Cas d'usage : Laboratoires, universités, industrie scientifique.

Résumé Express

  • HA : garantir un fonctionnement continu même en cas de panne
  • 3 piliers : Redondance, Failover, Monitoring
  • SLA : de 99 % (3,6 j/an) à 99,9999 % (31,5 s/an)
  • Clustering : regrouper plusieurs serveurs pour HA + répartition de charge + tolérance aux pannes
  • Active-Passive : un actif, un en veille (simple, sous-utilisation)
  • Active-Active : tous actifs, remplacements automatiques (performant, complexe)
  • Load Balancing : distribution des requêtes via load balancer
  • Failover Cluster : stockage partagé, reprise de services automatique
  • Grid Cluster : calcul distribué intensif
  • HA virtualisée : Proxmox/VMware, redémarrage auto des VM sur un autre hôte
