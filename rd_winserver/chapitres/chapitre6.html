Module 6

Haute Disponibilité

Le serveur principal de facturation tombe un lundi matin. 200 employés bloqués. Blaise sait que, sans redondance, chaque minute d'arrêt coûte des milliers d'euros. Voyons comment éviter ce cauchemar.

C'est quoi la Haute Disponibilité ?

La haute disponibilité (HA) c'est la capacité d'un système à rester opérationnel malgré les pannes. L'objectif : le service ne s'arrête jamais — ou presque.

Pense à un hôpital. Il a des générateurs de secours (redondance), des chirurgiens de garde (failover), et des moniteurs qui alertent (monitoring). Si un système tombe, un autre prend le relais immédiatement. Un hôpital ne peut pas se permettre de « redémarrer ».

La HA repose sur 3 piliers :

PilierRôleExemple Windows Server
RedondanceDupliquer les composants critiques2 Domain Controllers, 2 serveurs DNS
FailoverBasculer automatiquement vers le backupFailover Clustering
MonitoringDétecter les pannes en temps réelEvent Viewer, SCOM, Nagios

Mesurer la Disponibilité : les SLA

Le SLA (Service Level Agreement) exprime le temps de disponibilité garanti en pourcentage :

DisponibilitéSurnomTemps d'arrêt max / anType de service
99%« Deux neufs »3 jours 15hSite web personnel
99.9%« Trois neufs »8h 45minApplication métier
99.99%« Quatre neufs »52 minE-commerce, banque
99.999%« Cinq neufs »5 min 15sUrgences, aviation
99.9999%« Six neufs »31 secondesNucléaire, spatial
⚠️ Chaque « neuf » supplémentaire coûte exponentiellement plus cher. Passer de 99.9% à 99.99% peut multiplier le budget infrastructure par 10.

Types de Clusters

Un cluster = un groupe de serveurs qui travaillent ensemble comme un seul système.

1. Active-Passive (Failover simple)

Un nœud actif, un nœud en veille. Si l'actif tombe, le passif prend le relais.

C'est un pilote et un copilote. Le copilote est là « au cas où ». Il ne fait rien tant que le pilote va bien. Simple mais le copilote est « payé à ne rien faire ».
✅ Avantages❌ Inconvénients
Simple à mettre en placeSous-utilisation du nœud passif
Fiable et éprouvéPas d'augmentation de performance

2. Active-Active

Tous les nœuds traitent des requêtes. Si un tombe, les autres absorbent la charge.

Comme deux caisses ouvertes au supermarché. Les deux servent des clients. Si l'une ferme, l'autre gère toute la file — c'est plus lent mais ça marche.
✅ Avantages❌ Inconvénients
Utilisation optimale des ressourcesPlus complexe à configurer
Meilleures performances globalesRisque de surcharge si un nœud tombe

3. Network Load Balancing (NLB)

Distribue le trafic réseau entre plusieurs serveurs pour répartir la charge.

Un aiguilleur à l'entrée d'un parking : « Vous, niveau 1. Vous, niveau 2. » Chaque niveau est un serveur, et l'aiguilleur répartit les voitures équitablement.

Idéal pour : serveurs web, portails intranet, services sans état (stateless).

4. Failover Cluster avec stockage partagé

Les nœuds partagent un stockage SAN. Un seul nœud accède aux données à la fois.

Cas d'usage : SQL Server, serveurs de fichiers, Hyper-V.

Tableau comparatif

TypeNœuds actifsStockageCas d'usage
Active-Passive1Partagé (SAN)SQL, fichiers critiques
Active-ActiveTousPartagé ou répliquéApps distribuées
NLBTousPas de partage requisWeb, portails
Failover ClusterVariableSAN obligatoireSQL, Hyper-V, fichiers

Failover Clustering sous Windows Server

Prérequis

  • Minimum 2 serveurs Windows Server (même version)
  • Stockage partagé (SAN iSCSI, Fibre Channel, ou Storage Spaces)
  • Réseau dédié pour le heartbeat (communication inter-nœuds)
  • Domaine Active Directory (tous les nœuds dans le même domaine)

Installation

# Sur chaque nœud du cluster
Install-WindowsFeature -Name Failover-Clustering -IncludeManagementTools

# Valider la configuration (OBLIGATOIRE avant création)
Test-Cluster -Node "SRV-NODE1","SRV-NODE2"

# Créer le cluster
New-Cluster -Name "CLUSTER-PROD" `
    -Node "SRV-NODE1","SRV-NODE2" `
    -StaticAddress 192.168.1.200

# Vérifier l'état du cluster
Get-Cluster
Get-ClusterNode
💡 Test-Cluster génère un rapport HTML détaillé qui vérifie : réseau, stockage, configuration, compatibilité. Ne saute JAMAIS cette étape — Microsoft peut refuser le support si le test n'a pas été validé.

Quorum : le vote du cluster

Le quorum détermine combien de nœuds doivent être actifs pour que le cluster fonctionne :

ModeFonctionnementUsage
Node MajorityMajorité de nœuds vivantsCluster à 3+ nœuds
Node + Disk WitnessMajorité + disque témoinCluster à 2 nœuds
Node + File Share WitnessMajorité + partage réseauCluster multi-sites
Cloud WitnessTémoin dans AzureCluster hybride
Le quorum, c'est comme un vote dans un conseil d'administration. Si la majorité des membres est présente, les décisions sont valides. Sinon, on ne peut rien faire. Le « witness » (témoin) est un vote supplémentaire pour départager en cas d'égalité.

NLB (Network Load Balancing)

# Installer NLB sur chaque serveur
Install-WindowsFeature -Name NLB -IncludeManagementTools

# Créer un nouveau cluster NLB
New-NlbCluster -InterfaceName "Ethernet" `
    -ClusterPrimaryIP 192.168.1.250 `
    -SubnetMask 255.255.255.0 `
    -ClusterName "WEB-NLB" `
    -OperationMode Multicast

# Ajouter un deuxième nœud
Add-NlbClusterNode -InterfaceName "Ethernet" `
    -NewNodeInterface "Ethernet" `
    -NewNodeName "SRV-WEB2"

# Vérifier
Get-NlbCluster
⚠️ NLB vs Failover Cluster : Ne JAMAIS utiliser NLB pour des services qui écrivent sur disque (SQL, fichiers). NLB est pour les services stateless (web, portail, API). Pour le reste, utilisez Failover Clustering.

Réplication

La réplication synchronise les données entre serveurs pour la redondance :

TypeFonctionnementLatencePerte de données
SynchroneÉcriture confirmée quand les 2 copies sont faitesPlus élevéeAucune (RPO = 0)
AsynchroneÉcriture confirmée immédiatement, copie en arrière-planFaiblePossible (RPO > 0)

Storage Replica (Windows Server 2022)

# Installer Storage Replica
Install-WindowsFeature -Name Storage-Replica -IncludeManagementTools

# Créer un partenariat de réplication synchrone
New-SRPartnership -SourceComputerName "SRV-PARIS" `
    -SourceRGName "RG-Paris" `
    -SourceVolumeName "D:" `
    -SourceLogVolumeName "L:" `
    -DestinationComputerName "SRV-LYON" `
    -DestinationRGName "RG-Lyon" `
    -DestinationVolumeName "D:" `
    -DestinationLogVolumeName "L:" `
    -ReplicationMode Synchronous

Résumé du Module

  • HA = Redondance + Failover + Monitoring
  • SLA : chaque « neuf » supplémentaire = coût ×10
  • Active-Passive : simple mais sous-utilisé / Active-Active : optimal mais complexe
  • NLB pour les services stateless, Failover Cluster pour le reste
  • Quorum : le vote qui maintient le cluster cohérent
  • Test-Cluster = obligatoire avant toute création
  • Réplication synchrone (RPO=0) vs asynchrone (plus rapide)
