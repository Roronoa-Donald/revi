Entraînement SQL Server : Système E-commerce (Mode Sombre Avancé)

Ce document contient un grand script de préparation suivi d'exercices progressifs ciblant spécifiquement la résolution de problèmes réels d'E-commerce. L'accent est mis sur les procédures, triggers, et de nombreux cas de transactions avancées (dirty read, non-repeatable read, phantom read).

    CategoryID INT IDENTITY(1,1) PRIMARY KEY,
    CategoryName VARCHAR(100) NOT NULL,
    Description NVARCHAR(255)
);

-- 2. Table Products
CREATE TABLE Products (
    ProductID INT IDENTITY(1,1) PRIMARY KEY,
    CategoryID INT NOT NULL,
    ProductName NVARCHAR(150) NOT NULL,
    Price DECIMAL(10, 2) NOT NULL,
    StockQuantity INT NOT NULL DEFAULT 0,
    IsActive BIT DEFAULT 1,
    CONSTRAINT FK_Products_Categories FOREIGN KEY (CategoryID) REFERENCES Categories(CategoryID)
);

-- 3. Table Customers
CREATE TABLE Customers (
    CustomerID INT IDENTITY(1,1) PRIMARY KEY,
    FirstName NVARCHAR(50) NOT NULL,
    LastName NVARCHAR(50) NOT NULL,
    Email NVARCHAR(100) UNIQUE NOT NULL,
    RegistrationDate DATETIME DEFAULT GETDATE(),
    Status VARCHAR(20) DEFAULT 'Active'
);

-- 4. Table Orders
CREATE TABLE Orders (
    OrderID INT IDENTITY(1,1) PRIMARY KEY,
    CustomerID INT NOT NULL,
    OrderDate DATETIME DEFAULT GETDATE(),
    TotalAmount DECIMAL(12, 2) NOT NULL DEFAULT 0,
    OrderStatus VARCHAR(30) DEFAULT 'Pending', -- Pending, Shipped, Cancelled
    CONSTRAINT FK_Orders_Customers FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID)
);

-- 5. Table OrderItems
CREATE TABLE OrderItems (
    OrderItemID INT IDENTITY(1,1) PRIMARY KEY,
    OrderID INT NOT NULL,
    ProductID INT NOT NULL,
    Quantity INT NOT NULL,
    UnitPrice DECIMAL(10, 2) NOT NULL,
    CONSTRAINT FK_OrderItems_Orders FOREIGN KEY (OrderID) REFERENCES Orders(OrderID),
    CONSTRAINT FK_OrderItems_Products FOREIGN KEY (ProductID) REFERENCES Products(ProductID)
);

FROM Orders O
INNER JOIN Customers C ON O.CustomerID = C.CustomerID
INNER JOIN OrderItems OI ON O.OrderID = OI.OrderID

WHERE O.OrderID = 1;

SELECT P.ProductName 
FROM Products P
LEFT JOIN OrderItems OI ON P.ProductID = OI.ProductID
WHERE OI.OrderItemID IS NULL;

SELECT P.ProductName

Exercice 3.4 : Jointure à regroupement (CA par Catégorie) JOINS

SELECT C.CategoryName, SUM(OI.Quantity * OI.UnitPrice) AS ChiffreAffaires
FROM Categories C
INNER JOIN Products P ON C.CategoryID = P.CategoryID
INNER JOIN OrderItems OI ON P.ProductID = OI.ProductID

5.1 Procédures Stockées (3 Exercices)

Exercice P1 : Sp_UpdateOrderStatus PROCEDURE

Crée une procédure sp_UpdateOrderStatus prenant en paramètre @OrderID et @NewStatus. Elle doit modifier la commande. Si l'ID n'existe pas, renvoie une erreur personnalisée avec THROW.

CREATE PROCEDURE sp_UpdateOrderStatus
    @OrderID INT,
    @NewStatus VARCHAR(30)
AS
BEGIN
    SET NOCOUNT ON;
    
    IF NOT EXISTS (SELECT 1 FROM Orders WHERE OrderID = @OrderID)
    BEGIN
        THROW 50001, 'Erreur : Commande introuvable.', 1;
    END

    UPDATE Orders
    SET OrderStatus = @NewStatus
    WHERE OrderID = @OrderID;
    
    PRINT 'Statut mis à jour.';
END;
GO
Exercice P2 : Liste de commandes d'un client PROCEDURE

Crée une procédure sp_GetCustomerHistory avec comme paramètre @Email. Elle doit retourner une table listant l'ID de commande, la Date, et le Total, uniquement pour ce client.

CREATE PROCEDURE sp_GetCustomerHistory
    @Email NVARCHAR(100)
AS
BEGIN
    SET NOCOUNT ON;

    SELECT O.OrderID, O.OrderDate, O.TotalAmount, O.OrderStatus
    FROM Orders O
    INNER JOIN Customers C ON O.CustomerID = C.CustomerID
    WHERE C.Email = @Email
    ORDER BY O.OrderDate DESC;
END;
GO
Exercice P3 : Application de remise massive PROCEDURE

Crée sp_ApplyGlobalDiscount (@CategoryID INT, @DiscountRate DECIMAL(5,2)). Elle met à jour le Price des produits de la catégorie ciblée (Ex: Price * (1 - @DiscountRate / 100)). Retourne le nombre de lignes modifiées (@@ROWCOUNT).

CREATE PROCEDURE sp_ApplyGlobalDiscount
    @CategoryID INT,
    @DiscountRate DECIMAL(5,2)
AS
BEGIN
    SET NOCOUNT ON;

    UPDATE Products
    SET Price = Price * (1.0 - (@DiscountRate / 100.0))
    WHERE CategoryID = @CategoryID;

    DECLARE @Lignes INT = @@ROWCOUNT;
    PRINT Cast(@Lignes as Varchar) + ' produits mis à jour.';
END;
GO

5.2 Triggers (3 Exercices)

Exercice TR1 : Audit de changement de Prix TRIGGER

Crée d'abord une table PriceHistory. Ensuite, associe un trigger AFTER UPDATE sur Products : S'il y a un changement de prix, on log l'ancien prix (DELETED) et le nouveau (INSERTED).

CREATE TABLE PriceHistory (
    LogID INT IDENTITY(1,1), ProductID INT, OldPrice DECIMAL(10,2), NewPrice DECIMAL(10,2), ChangeDate DATETIME DEFAULT GETDATE()
);
GO
CREATE TRIGGER trg_AuditPriceChange ON Products
AFTER UPDATE AS
BEGIN
    IF UPDATE(Price)
    BEGIN
        INSERT INTO PriceHistory (ProductID, OldPrice, NewPrice)
        SELECT i.ProductID, d.Price, i.Price 
        FROM inserted i INNER JOIN deleted d ON i.ProductID = d.ProductID
        WHERE i.Price <> d.Price;
    END
END;
GO
Exercice TR2 : Blocage de suppression TRIGGER

Crée un trigger INSTEAD OF DELETE sur Customers. Si le client a le statut 'Active', bloque la suppression et lève une erreur. S'il n'est pas actif, autorise la suppression.

CREATE TRIGGER trg_PreventDeleteActiveCustomer
ON Customers
INSTEAD OF DELETE
AS
BEGIN
    IF EXISTS (SELECT 1 FROM deleted WHERE Status = 'Active')
    BEGIN
        THROW 50000, 'Impossible de supprimer un client Actif. Passez-le d''abord en inactif.', 1;
    END
    ELSE
    BEGIN
        DELETE FROM Customers
        WHERE CustomerID IN (SELECT CustomerID FROM deleted);
    END
END;
GO
Exercice TR3 : Restitution de stock automatique TRIGGER

Crée un trigger AFTER UPDATE sur Orders. Si le statut passe spécifiquement de n'importe quoi à 'Cancelled', va lire OrderItems et remets les quantités dans Products automatiquement.

CREATE TRIGGER trg_UpdateStockOnOrderCancel
ON Orders
AFTER UPDATE
AS
BEGIN
    -- On vérifie si la colonne OrderStatus a été modifiée
    IF UPDATE(OrderStatus)
    BEGIN
        -- On boucle (via un join) sur les commandes qui Viennent de passer à "Cancelled"
        UPDATE P
        SET P.StockQuantity = P.StockQuantity + OI.Quantity
        FROM Products P
        INNER JOIN OrderItems OI ON P.ProductID = OI.ProductID
        INNER JOIN inserted i ON OI.OrderID = i.OrderID
        INNER JOIN deleted d ON i.OrderID = d.OrderID
        WHERE i.OrderStatus = 'Cancelled' AND d.OrderStatus <> 'Cancelled';
    END
END;
GO

5.3 Transactions & Niveaux d'Isolation (4 Exercices)

Exercice TX1 : Transaction Classique TRY..CATCH TRANSACTION

Crée une procédure sp_Checkout qui ouvre une transaction : elle insert une Order, insert son OrderItem, et diminue le stock. Si la Qté < Stock, THROW erreur et fait un ROLLBACK.

CREATE PROCEDURE sp_Checkout (@CustID INT, @ProdID INT, @Qty INT)
AS
BEGIN
    BEGIN TRY
        BEGIN TRAN;
        
        DECLARE @Stock INT; SELECT @Stock = StockQuantity FROM Products WHERE ProductID = @ProdID;
        IF @Stock < @Qty THROW 50000, 'Stock insuffisant', 1;

        -- Insertion entete
        INSERT INTO Orders (CustomerID, TotalAmount) VALUES (@CustID, 0);
        DECLARE @OrdID INT = SCOPE_IDENTITY();
        
        -- Detaille et MAJ Stock
        INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice) VALUES (@OrdID, @ProdID, @Qty, 100);
        UPDATE Products SET StockQuantity = StockQuantity - @Qty WHERE ProductID = @ProdID;

        COMMIT TRAN;
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0 ROLLBACK TRAN;
        THROW;
    END CATCH
END;
GO
Exercice TX2 : Dirty Read (Read Uncommitted) ISOLATION

Problème : Transaction A modifie un stock mais ne valide pas (WAITFOR). Transaction B vient faire un rapport financier rapide des stocks.

Montre quelles commandes tapées dans SSMS (Fenêtre 1 et Fenêtre 2) permettent à la Fenêtre 2 de lire le stock pendant la maj de la fenêtre 1 (permettant un "Dirty Read").

-- FENÊTRE 1 (SSMS):
BEGIN TRAN;
UPDATE Products SET StockQuantity = 999 WHERE ProductID = 1;
WAITFOR DELAY '00:00:10'; -- Simule un processus très lent
ROLLBACK; -- Finalement, erreur on annule.

-- FENÊTRE 2 (Dans un 2eme onglet, pendant les 10 secondes) :
-- Par défaut, T-SQL (Read Committed) se bloquera. 
-- Pour autoriser la lecture "sale", on utilise READ UNCOMMITTED (ou as (NOLOCK))
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
BEGIN TRAN;
SELECT StockQuantity FROM Products WHERE ProductID = 1; -- Lira '999' (FAUX, puisque ça va être rollback!)
COMMIT;
Exercice TX3 : Non-Repeatable Read & REPEATABLE READ ISOLATION

Problème : Un audit lit le Prix d'un Produit pour un calcul complexe (T1). Pendant son calcul (T2), un admin web update le Prix du produit. A la fin du calcul, T1 relit le Prix, mais il a changé !

Comment forcer le système (avec le niveau d'isolation) à empêcher (bloquer) l'admin T2 de modifier pendant que T1 lit le prix ?

-- FENÊTRE 1 (L'audit, qui bloque pour empêcher la modification de la ligne lue) :
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
BEGIN TRAN;
SELECT Price FROM Products WHERE ProductID = 1; -- Pose un verrou de lecture persistant sur cette ligne !
WAITFOR DELAY '00:00:10';
SELECT Price FROM Products WHERE ProductID = 1; -- C'est garanti que c'est le même prix.
COMMIT;

-- FENÊTRE 2 (L'Admin, qui va être bloqué) :
BEGIN TRAN;
-- Cette requête va tourner dans le vide et attendre que FENÊTRE 1 fasse son COMMIT
UPDATE Products SET Price = 80 WHERE ProductID = 1; 
COMMIT;
Exercice TX4 : Phantom Read & SERIALIZABLE ISOLATION

Problème : Une comptabilité (T1) compte les produits de la catégorie 1 (Ex: trouve 3 produits). Avant de finir, un gérant (T2) rajoute un nouveau produit catégorie 1. Si T1 recompte, hop, il y en a 4 ! C'est le "Phantom Read".

Quel est le niveau de protection maximal pour geler totalement une "plage de critères de recherche" (qui bloque les INSERT pour la Catégorie 1 pendant T1) ?

-- FENÊTRE 1 (Compteur strict) :
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN TRAN;
SELECT COUNT(*) FROM Products WHERE CategoryID = 1; -- Pose un verrou de plage sur la table et les index !
WAITFOR DELAY '00:00:10';
SELECT COUNT(*) FROM Products WHERE CategoryID = 1;
COMMIT;

-- FENÊTRE 2 (Gérant qui veut insérer) :
BEGIN TRAN;
-- Bloqué ! L'insertion d'un CategoryID = 1 est interdite tant que FENETRE 1 n'est pas COMMIT.
INSERT INTO Products (CategoryID, ProductName, Price) VALUES (1, 'Disque Dur', 50);
COMMIT;
