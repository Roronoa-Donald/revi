1. Le Gardien du Temple

Une base de données n'est pas une simple poubelle où l'on jette des données en vrac. C'est un coffre-fort organisé.

Imaginez que vous gérez les comptes bancaires. Pouvez-vous accepter une transaction sans montant ? Un client sans nom ? Deux clients avec le même identifiant ?
Non. C'est là qu'interviennent les Contraintes. Ce sont les gardes du corps de vos données.

2. L'Identité Unique (Primary Key)

Chaque ligne est unique

Le Problème des Homonymes

Imaginez deux clients nommés "Thomas Dupont". Si l'un d'eux appelle pour résilier, comment savoir lequel c'est ?

  • SANS PK Thomas Dupont (Paris) / Thomas Dupont (Lyon) -> Confusion !
  • AVEC PK Client #1024 vs Client #4598 -> Clair !

La Clé Primaire (PK) est un numéro unique (souvent un ID) qui identifie formellement une ligne.

3. Le Lien Familial (Foreign Key)

Les tables ne vivent pas en solitaire. Elles discutent entre elles. Comment dire que la "Commande #555" appartient au "Client #1024" ?

Relation

La Clé Étrangère (FK) est une colonne dans la table enfant qui contient la carte de visite du parent.
Règle d'or : Vous ne pouvez pas créer une commande pour le client #9999 si le client #9999 n'existe pas !

Zoom : Le Drame de la Suppression

Que se passe-t-il si j'efface le Papa ?

C'est la question critique de l'intégrité référentielle. Si je supprime le Client #1024, que deviennent ses commandes ?

RESTRICT (Par défaut)

SQL Server vous dit STOP. "Je ne peux pas supprimer ce client car il a des commandes actives. Occupe-toi des commandes d'abord."

DELETE statement conflicted with REFERENCE constraint...

CASCADE (Dangereux)

SQL Server dit OK. Il supprime le client ET automatiquement toutes ses commandes liées. Le ménage par le vide.

(124 rows affected) // Oups, tout est parti !

1. Clé Primaire

Solution

Créez une table 'Etudiants' avec 'matricule' (VARCHAR 10) comme clé primaire.

CREATE TABLE Etudiants (
  matricule VARCHAR(10) PRIMARY KEY,
  nom VARCHAR(50)
);

2. Clé Étrangère

Solution

Reliez 'Commande' à 'Client' (id) via client_id.

ALTER TABLE Commande ADD CONSTRAINT fk_cli
FOREIGN KEY (client_id) REFERENCES Client(id);

3. NOT NULL

Solution

Forcez l'email à ne jamais être vide lors de la création.

email VARCHAR(100) NOT NULL

4. UNIQUE

Solution

Assurez-vous qu'on ne puisse pas avoir deux fois le même email.

ALTER TABLE Users ADD CONSTRAINT uq_email UNIQUE (email);

5. CHECK

Solution

Interdire un âge négatif.

CHECK (age >= 0)
