1. Le Pacte du Tout-ou-Rien

Imaginez un virement bancaire : vous débitez 500€ du compte A... mais l'électricité coupe avant de créditer le compte B. L'argent a disparu !

En SQL, les transactions garantissent que cette catastrophe n'arrive jamais. C'est le principe ACID : soit TOUT réussit, soit RIEN ne change.

2. ACID — Les 4 Garanties

Atomicité

Tout ou rien. Si une opération échoue, toutes les modifications sont annulées.

« Le virement est complet ou n'a jamais eu lieu. »

Cohérence

La base passe d'un état valide à un autre état valide. Les contraintes sont respectées.

« Le solde ne peut jamais être négatif si la contrainte existe. »

Isolation

Les transactions concurrentes ne se voient pas mutuellement. Chacune croit être seule.

« Deux guichetiers ne modifient pas le même solde en même temps. »

Durabilité

Une fois COMMIT fait, les données survivent même en cas de crash ou coupure de courant.

« L'écriture est gravée sur disque, pas seulement en mémoire. »

3. BEGIN / COMMIT / ROLLBACK

BEGIN TRANSACTION

UPDATE Comptes SET solde = solde - 500 WHERE id = 1
UPDATE Comptes SET solde = solde + 500 WHERE id = 2

IF @@ERROR = 0
    COMMIT -- Tout est bon, on valide
ELSE
    ROLLBACK -- Problème ! On annule tout
BEGIN TRAN

Démarre la transaction

COMMIT

Valide définitivement

ROLLBACK

Annule tout

4. SAVEPOINT — Point de Sauvegarde

Un SAVEPOINT permet d'annuler partiellement une transaction sans perdre tout le travail.

BEGIN TRANSACTION
  INSERT INTO Commandes VALUES(1, 'Laptop', 1200)

  SAVE TRANSACTION point1

  INSERT INTO Commandes VALUES(2, 'Souris', 25)
  -- Oups, erreur sur cette insertion
  ROLLBACK TRANSACTION point1 -- Annule seulement la souris

COMMIT -- Le laptop est validé !
Analogie : C'est comme un jeu vidéo — le SAVEPOINT est votre checkpoint. Si vous mourez, vous ne recommencez pas du début.

5. Niveaux d'Isolation

L'isolation détermine ce qu'une transaction voit quand d'autres travaillent en même temps :

Niveau Dirty Read Non-Repeatable Phantom Performance
READ UNCOMMITTED ⚠️ Oui ⚠️ Oui ⚠️ Oui ⚡⚡⚡
READ COMMITTED ✅ Non ⚠️ Oui ⚠️ Oui ⚡⚡
REPEATABLE READ ✅ Non ✅ Non ⚠️ Oui ⚡
SERIALIZABLE ✅ Non ✅ Non ✅ Non 🐌
Dirty Read : Lire une donnée modifiée par une autre transaction non encore validée.
Non-Repeatable Read : Deux lectures identiques dans la même transaction donnent des résultats différents.
Phantom Read : Des lignes apparaissent/disparaissent entre deux lectures (INSERT/DELETE par un tiers).
SET TRANSACTION ISOLATION LEVEL READ COMMITTED
BEGIN TRANSACTION
  -- Vos opérations ici
COMMIT

6. Deadlock — L'Impasse

Un deadlock survient quand deux transactions s'attendent mutuellement :

Transaction A
1. Verrouille Table1 ✅
2. Attend Table2... 🔒⏳
Transaction B
1. Verrouille Table2 ✅
2. Attend Table1... 🔒⏳
Résultat : SQL Server détecte l'impasse et tue la transaction la « moins coûteuse » (deadlock victim).
Prévention : Accédez toujours aux tables dans le même ordre, gardez les transactions courtes, évitez les verrous inutiles.
