1. L'Usine à Requêtes

Imaginez que vous devez vérifier le stock d'un produit 50 fois par jour. Allez-vous réécrire la même requête à chaque fois ? Non !

Une procédure stockée, c'est une recette : vous l'écrivez une fois, vous la rangez dans la base de données, et ensuite vous l'exécutez d'un simple EXEC. C'est le passage de l'artisanat à l'industrialisation.

2. Créer une Procédure

CREATE PROCEDURE sp_GetClient
    @id INT
AS
BEGIN
    SELECT nom, email, date_inscription
    FROM Clients
    WHERE id = @id
END
Convention : Préfixez vos procédures par sp_ (stored procedure) ou usp_ (user stored procedure) pour les distinguer des procédures système.

3. Paramètres IN et OUT

Paramètre IN (entrée)

Valeur envoyée à la procédure (par défaut).

CREATE PROCEDURE sp_Note
  @etudiant_id INT,
  @matiere VARCHAR(50)
AS ...

Paramètre OUTPUT

Valeur retournée par la procédure.

CREATE PROCEDURE sp_Count
  @total INT OUTPUT
AS
BEGIN
  SELECT @total = COUNT(*)
  FROM Clients
END
Appel avec OUTPUT :
DECLARE @resultat INT
EXEC sp_Count @total = @resultat OUTPUT
PRINT @resultat

4. Exécuter, Modifier, Supprimer

EXEC sp_GetClient 5

Exécuter avec paramètre positionnel

ALTER PROCEDURE

Modifier une procédure existante

DROP PROCEDURE

Supprimer définitivement

Paramètre nommé vs positionnel :
EXEC sp_GetClient 5 → positionnel (ordre des paramètres)
EXEC sp_GetClient @id = 5 → nommé (plus lisible)

5. Gestion d'Erreurs dans une Procédure

CREATE PROCEDURE sp_Virement
  @from INT, @to INT, @montant DECIMAL(10,2)
AS
BEGIN
  BEGIN TRY
    BEGIN TRANSACTION
      UPDATE Comptes SET solde = solde - @montant WHERE id = @from
      UPDATE Comptes SET solde = solde + @montant WHERE id = @to
    COMMIT
  END TRY
  BEGIN CATCH
    ROLLBACK
    PRINT ERROR_MESSAGE()
  END CATCH
END
Best Practice : Combinez toujours TRY/CATCH + BEGIN TRAN/COMMIT/ROLLBACK dans vos procédures pour une robustesse maximale.

6. Pourquoi utiliser des Procédures ?

Sécurité

Les utilisateurs exécutent la procédure sans accéder directement aux tables. Protection contre l'injection SQL.

Performance

Le plan d'exécution est compilé et mis en cache à la première exécution, accélérant les appels suivants.

Maintenance

Modifiez la logique en un seul endroit (ALTER PROCEDURE) sans toucher à l'application.

Réseau

Un seul appel EXEC traverse le réseau, au lieu d'envoyer de longues requêtes texte.
