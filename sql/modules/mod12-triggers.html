1. L'Effet Papillon de la Base de Données

Un client est supprimé de la base. Automatiquement, un email de confirmation est envoyé, l'historique est archivé, et le manager est notifié. Personne n'a écrit de code pour ça — c'est le trigger qui a tout déclenché.

Un trigger, c'est un piège invisible posé sur une table. Dès qu'une action se produit (INSERT, UPDATE, DELETE), le trigger se déclenche automatiquement.

Attention : Les triggers sont puissants mais dangereux. Un trigger mal conçu peut ralentir drastiquement votre base ou créer des boucles infinies (un trigger qui déclenche un autre trigger...).

2. AFTER — Après l'Action

Le trigger AFTER s'exécute après que l'opération a réussi.

CREATE TRIGGER trg_AuditInsert
ON Clients
AFTER INSERT
AS
BEGIN
    INSERT INTO AuditLog (action, table_name, date_action)
    SELECT 'INSERT', 'Clients', GETDATE()
    FROM inserted
END
AFTER INSERT

Après un ajout

AFTER UPDATE

Après une modification

AFTER DELETE

Après une suppression

3. Les Tables Magiques : inserted & deleted

SQL Server crée automatiquement deux tables virtuelles dans chaque trigger :

Opération inserted deleted
INSERT ✅ Nouvelles lignes — vide
UPDATE ✅ Nouvelles valeurs ✅ Anciennes valeurs
DELETE — vide ✅ Lignes supprimées
Astuce UPDATE : Pour savoir quelle colonne a changé, comparez inserted.colonne avec deleted.colonne.

4. INSTEAD OF — Interception

INSTEAD OF remplace l'opération originale. Idéal pour le « soft delete » :

CREATE TRIGGER trg_SoftDelete
ON Clients
INSTEAD OF DELETE
AS
BEGIN
    -- Au lieu de supprimer, on marque comme inactif
    UPDATE Clients
    SET actif = 0, date_suppression = GETDATE()
    WHERE id IN (SELECT id FROM deleted)
END
Soft Delete : Les données ne sont jamais vraiment supprimées — elles sont marquées inactives. Utile pour l'audit, la conformité RGPD, et la récupération de données.

5. Cas d'Usage Réels

Audit automatique

Logger qui a modifié quoi, quand, avec AFTER UPDATE sur les tables sensibles.

Validation métier

Vérifier qu'un stock ne passe pas en négatif avec AFTER UPDATE + ROLLBACK.

Synchronisation

Mettre à jour une table dérivée quand la table source change.

Notifications

Insérer dans une table de notifications pour alerter les utilisateurs.

6. Pièges & Performance

À éviter
  • • Triggers récursifs (trigger qui en déclenche un autre)
  • • Logique métier complexe dans un trigger
  • • Envoi d'emails depuis un trigger
  • • Triggers sur tables à fort volume (millions d'INSERT/sec)
Bonnes pratiques
  • • Garder les triggers courts et simples
  • • Documenter chaque trigger
  • • Préférer les procédures stockées si possible
  • • Tester avec des volumes réalistes
-- Désactiver un trigger temporairement
DISABLE TRIGGER trg_AuditInsert ON Clients

-- Réactiver
ENABLE TRIGGER trg_AuditInsert ON Clients

-- Supprimer
DROP TRIGGER trg_AuditInsert
