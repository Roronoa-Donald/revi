Module 1 — UI/UX & Design Responsive : Introduction

UX (User Experience — Expérience Utilisateur) : Ce que l'utilisateur ressent par rapport à un produit — présentation, clarté du contenu, satisfaction générale. L'UX est invisible mais déterminante.
UI (User Interface — Interface Utilisateur) : Ce que l'utilisateur voit — typographie, couleurs, menus, boutons, icônes, images. L'UI est la partie visuelle et tangible.
Responsive Design : Capacité d'une interface à s'adapter automatiquement à la taille de l'écran (mobile, tablette, desktop, large desktop).

UX vs UI — La différence clé

UX (Ressenti)

  • Facilité d'utilisation
  • Parcours utilisateur fluide
  • Satisfaction globale
  • Résolution de problèmes
  • Rapidité des actions

UI (Visuel)

  • Choix des couleurs
  • Typographie et taille
  • Disposition des boutons
  • Icônes et images
  • Animations et transitions
Si un produit est un restaurant : l'UX c'est la qualité du service, la rapidité, l'ambiance — l'UI c'est la présentation de l'assiette, la décoration, le menu.
💡 Retenir : Un bon UI sans bon UX = un beau produit inutilisable. Un bon UX sans bon UI = un produit fonctionnel mais laid. Il faut les deux.

Module 2 — CCU : Conception Centrée Utilisateur

CCU (Conception Centrée Utilisateur) : Modèle de conception en 4 étapes centré sur les besoins réels de l'utilisateur. Principe clé : les intentions du concepteur ne comptent pas — seul l'utilisateur compte.

Acteurs impliqués : Designers, Développeurs, Programmeurs, Ergonomes.

Les 4 étapes de la CCU

1

Comprendre l'utilisateur

Identifier qui est l'utilisateur, ses besoins, ses frustrations, ses motivations. Définir le type de contenu à proposer en fonction de la cible.

3 phases de pré-conception :
1ère phase — Collecte et analyse des données
2ème phase — Conception Culturelle : définir l'organisation du contenu et comment on y accède
3ème phase — Parcours Utilisateur : comprendre comment l'utilisateur mène à bien sa transaction
2

Concevoir

Idée → Croquis → Wireframe → Maquette. On passe de l'abstrait au concret de manière progressive.

3

Prototyper

Créer un prototype interactif avec des outils comme Figma, Adobe XD. Le prototype permet de simuler les interactions avant le développement.

4

Tester et Améliorer

Tests utilisateur, analyse des retours, corrections. Le cycle recommence si nécessaire (processus itératif).

Pensée du concepteur vs Pensée de l'utilisateur

🧑‍💻 Concepteur pense

  • Acheter un produit
  • Trouver un site joli
  • Apprendre à utiliser l'app

👤 Utilisateur pense

  • Résoudre un problème
  • Obtenir satisfaction
  • Interface simple et rapide

Gain de temps avec le prototypage

❌ Sans prototype

  • Design (16h) → Dev (30h) → Design (16h) → Dev (30h)
  • Total : ~92h avec corrections

✅ Avec prototype

  • Prototype (19h) → Design (4h) → Dev (31h)
  • Total : ~54h optimisé
Avantages de la CCU : Validation des étapes avant de continuer • Arbitrage des options pendant l'optimisation • Réduction du coût de déploiement • Moins de retours en arrière.

Module 3 — UX Research : Collecte de données & Persona

Méthodes de collecte de données

  • Faire un sondage
  • Analyse du coût de vie
  • Enquête sur le pouvoir d'achat
  • Observer les personnes autour de vous
  • Qualifier et sélectionner les participants
  • Recherche documentaire
  • Analyser les données de sites concurrents
  • Auto-tester l'application
  • Interviews utilisateurs

Les 4 étapes de l'UX Research

1

Collecte des données

Sondage, analyse du coût de vie, enquête sur le pouvoir d'achat.

2

Les motifs et les besoins

Motifs : Pas envie de se déplacer, recherche de plats exotiques, manque de temps.
Besoins : Commander facilement, suivi du trajet en temps réel.

3

Définir le Persona

Créer une représentation semi-fictive de l'utilisateur cible basée sur des données réelles.

4

Recommandations UX

L'application doit montrer le prix avant de cliquer • Commande en 2-3 clics • Restaurants proches en premier.

Qu'est-ce qu'un Persona ?

Persona : Représentation semi-fictive d'un utilisateur cible basée sur des données réelles. Il inclut : échelle de valeur, motivations, intérêts, ce qui le persuade, les médias sociaux qu'il fréquente.

Module 4 — Wireframes & Prototypage

Wireframe : Architecture de l'information — représentation schématique d'une interface sans design visuel final.

Niveaux de fidélité

📝 Basse fidélité (Low-fi)

  • Croquis simple, rapide
  • Pas de couleurs ni de design
  • Focus sur la structure
  • Fait à la main ou outils simples

🎨 Haute fidélité (High-fi)

  • Maquette détaillée
  • Design, couleurs, interactions
  • Proche du produit final
  • Fait avec Figma, Adobe XD

Design System

Un Design System est un ensemble de composants visuels réutilisables avec un thème commun. Il contient :
  • Règles d'effets : disposition, comportement, interaction
  • Composants communs partagés entre les équipes
  • Standards de design cohérents

Étapes du wireframing

1

Idée

Conception initiale, brainstorming.

2

Croquis manuel

Dessins rapides sur papier.

3

Wireframe basse fidélité

Structure de base sans design.

4

Maquette haute fidélité

Design détaillé avec couleurs et typographie.

5

Prototype interactif

Navigation simulée entre les écrans (Figma, Adobe XD).

6

Validation et correction

Tests, retours, itérations.

Module 5 — Design Responsive

Design Responsive : Méthode qui permet à une interface de s'adapter automatiquement à la taille de l'écran. C'est la philosophie de la conception moderne.

Deux approches

🖥️ Desktop-First

  • On conçoit d'abord pour le grand écran
  • Puis on réduit vers tablette et mobile
  • Utilise @media (max-width)

📱 Mobile-First (recommandé)

  • On conçoit d'abord pour le mobile
  • Puis on agrandit progressivement
  • Utilise @media (min-width)
⚠️ Mobile-First est l'approche recommandée car la majorité du trafic web est mobile. On commence par l'essentiel puis on enrichit.

Media Queries

Media Query : Point dans le code CSS où le style change pour s'adapter à la taille de l'écran. C'est le mécanisme technique du responsive design.
/* Mobile-First : styles de base pour mobile */ .container { display: flex; flex-direction: column; /* empilé sur mobile */ } /* À partir de 576px → Tablette */ @media (min-width: 576px) { .container { flex-direction: row; /* côte à côte */ } } /* Desktop-First : styles de base pour desktop */ @media (max-width: 576px) { .container { flex-direction: column; /* empilé sur mobile */ } }
Retenir :
• min-width → « à partir de X pixels » → utilisé en Mobile-First (on agrandit)
• max-width → « jusqu'à X pixels » → utilisé en Desktop-First (on réduit)

Module 6 — Organisation des éléments & Auto Layout (Figma)

Auto Layout : Fonctionnalité de Figma qui permet d'organiser automatiquement les éléments dans un cadre (frame). C'est l'équivalent visuel de Flexbox dans Figma.

Principes de l'Auto Layout

  • Direction : Verticale (↓) ou Horizontale (→) — comme flex-direction
  • Espacement (Gap) : Distance entre les éléments enfants
  • Padding : Espacement intérieur du conteneur (haut, droite, bas, gauche)
  • Alignement : Comment les éléments se positionnent dans l'axe principal et secondaire

Les 3 modes de redimensionnement

ModeDescriptionÉquivalent CSS
Hug Contents Le cadre s'adapte à la taille de son contenu — il « enveloppe » les éléments width: fit-content
Fill Container L'élément s'étire pour remplir tout l'espace disponible de son parent width: 100% / flex: 1
Fixed Une taille fixe en pixels — ne change pas width: 300px
⚠️ Très important pour l'examen :
• Hug = le parent s'adapte au contenu (comme un emballage-cadeau)
• Fill = l'enfant s'adapte au parent (comme de l'eau dans un verre)
• Fixed = taille rigide, ne bouge pas

Organisation dans un Auto Layout

Direction verticale (↓) — Les éléments s'empilent de haut en bas. Idéal pour : listes, cartes mobiles, menus verticaux.

Contraintes (Constraints)

Les contraintes définissent comment un élément se comporte quand son parent est redimensionné :
  • Left / Right / Top / Bottom : L'élément reste ancré à ce côté
  • Left and Right : L'élément s'étire horizontalement
  • Center : L'élément reste centré
  • Scale : L'élément se redimensionne proportionnellement
Auto Layout = Responsive dans Figma. C'est le mécanisme qui permet de créer des maquettes qui s'adaptent automatiquement, tout comme les media queries en CSS.

Module 7 — Réorganisation Responsive & Raisonnement UI

Réorganisation Responsive : Processus de réarrangement des éléments d'une interface lorsqu'on passe d'un écran large (desktop) à un écran étroit (mobile), ou inversement. C'est la logique responsive.

Principes de réorganisation

Règle N°1 — Empilement : Sur mobile, les éléments côte à côte (horizontal) deviennent empilés (vertical). C'est la transformation la plus courante.

Logique de raisonnement UI responsive

1

Identifier les blocs

Découper l'interface en blocs logiques : header, nav, contenu, sidebar, footer.

2

Définir la priorité

Quel bloc est le plus important ? Le contenu principal passe en premier sur mobile.

3

Choisir la direction

Sur mobile → vertical (column). Sur desktop → horizontal (row). C'est le changement fondamental.

4

Ajuster les tailles

Mobile : 100% width (Fill). Desktop : pourcentages ou fractions. Certains éléments peuvent être masqués.

5

Masquer / Réorganiser

Certains éléments secondaires (sidebar, sous-menus) sont masqués sur mobile ou déplacés plus bas.

Exemple complet : Layout Desktop → Mobile

Auto Layout et Responsive dans Figma

Dans Figma, la réorganisation responsive se fait avec :
  • Changer la direction de l'Auto Layout (horizontal → vertical)
  • Passer de Fixed à Fill Container pour que les éléments s'adaptent
  • Activer le Wrap pour que les éléments passent à la ligne
  • Masquer des éléments (la sidebar, par exemple)
  • Ajuster le padding et le gap (réduire les espacements)
❌ Erreur courante : Ne pas réorganiser les éléments pour mobile. Un layout 3 colonnes desktop affiché tel quel sur mobile = illisible et inutilisable.

Module 8 — Ergonomie

Ergonomie : Procédé par lequel on améliore la perception de l'utilisateur par rapport à l'application, en appliquant des principes psychologiques et cognitifs.

Les 3 lois fondamentales

Loi de Fitts

Plus une cible est grande et proche, plus elle est facile et rapide à atteindre.

Application : Augmenter la taille des boutons importants (Valider, Connexion, S'inscrire) et réduire leur distance par rapport à l'action courante.

Taille ↑ + Distance ↓ = Action rapide ✅

Loi de Hick

Plus il y a d'options, plus l'utilisateur met du temps à décider.

Application : Réduire le nombre de choix à l'écran. Éviter les menus avec 20 options. Simplifier les formulaires.

Nombre d'options ↑ = Temps de décision ↑ ❌

Loi de Jakob

Les utilisateurs préfèrent ce qu'ils connaissent déjà.

Application : Utiliser des patterns d'interface familiers. Menu hamburger (☰) en haut, panier en haut à droite, logo en haut à gauche.

Familiarité → Confort → Adoption plus rapide 🏠
La loi de Jakob explique pourquoi les sites e-commerce se ressemblent : les utilisateurs savent déjà naviguer. Innover c'est bien, dérouter c'est dangereux.

Module 9 — Accessibilité Web — WCAG

Accessibilité : Rendre les contenus web utilisables par tous, y compris les personnes en situation de handicap. Référentiel : WCAG (Web Content Accessibility Guidelines).

Les 4 principes WCAG (POCR)

P

Perceptible

Le contenu doit être visible et compréhensible par tous les sens. Texte alternatif, bon contraste des couleurs.

O

Opérable

L'utilisateur peut naviguer quelle que soit sa méthode : clavier, souris, voix. Focus visible obligatoire.

C

Compréhensible

Interface simple, labels clairs, messages d'erreur explicites et compréhensibles.

R

Robuste

Code propre et valide, attributs ARIA, compatible avec les lecteurs d'écran.

ARIA — Accessible Rich Internet Applications

ARIA : Attributs HTML qui rendent le contenu accessible aux technologies d'assistance (lecteurs d'écran). Exemples :
  • aria-label="Fermer" — donne un nom accessible à un bouton
  • role="navigation" — identifie la zone de navigation
  • aria-expanded="false" — indique si un menu est ouvert ou fermé
  • alt="Description de l'image" — texte alternatif pour les images

Outils de test d'accessibilité

Module 10 — Typographie & Espacement

Hiérarchie visuelle

La hiérarchie visuelle guide l'œil de l'utilisateur à travers le contenu. Elle repose sur :
  • Taille : H1 > H2 > H3 > paragraphe
  • Poids : Bold > Regular > Light
  • Couleur : Couleur d'accent > Texte principal > Texte secondaire
  • Espacement : Plus d'espace = plus d'importance
H1 — Titre Principal
H2 — Sous-titre
H3 — Section
Paragraphe — Texte courant, description, explication détaillée.

Espacement : margin, padding, gap

PropriétéRôleAnalogie
marginEspacement extérieur autour de l'élémentLa distance entre deux maisons
paddingEspacement intérieur dans l'élémentL'épaisseur des murs d'une maison
gapEspacement entre les éléments enfants (dans Flex/Grid)L'espace entre les rangs de sièges

Bonnes pratiques de lisibilité

  • Taille de police minimum : 16px pour le texte courant
  • Contraste suffisant entre texte et fond (ratio ≥ 4.5:1 selon WCAG)
  • Line-height (interligne) entre 1.5 et 1.8
  • Maximum 60-80 caractères par ligne pour le confort de lecture
  • Espacement cohérent : utiliser des multiples (4px, 8px, 16px, 24px, 32px...)
