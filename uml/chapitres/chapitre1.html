Module 1

Introduction à UML

Découvrez pourquoi modéliser, l'histoire d'UML et les 13 types de diagrammes du langage unifié de modélisation.

Pourquoi modéliser ?

Imaginez construire un immeuble sans plan d'architecte : on ne saurait ni par quoi commencer, ni comment les étages s'agencent. Un modèle UML est le plan de votre logiciel.

La modélisation est une activité fondamentale dans tout projet informatique. Elle permet de :

  • Visualiser la structure et le comportement du système
  • Communiquer entre les membres de l'équipe (développeurs, analystes, clients)
  • Documenter les décisions de conception
  • Détecter les erreurs tôt, avant même d'écrire du code
💡 Règle d'or : Un bon modèle simplifie la réalité en ne gardant que les aspects pertinents pour le problème à résoudre.

Histoire d'UML

UML (Unified Modeling Language) est né de la fusion de trois méthodes concurrentes dans les années 1990 :

AuteurMéthodeContribution
Grady BoochMéthode BoochConception orientée objet
James RumbaughOMTModélisation des données
Ivar JacobsonOOSECas d'utilisation

Chronologie

  • 1994-1996 — Booch, Rumbaugh et Jacobson fusionnent leurs méthodes chez Rational Software
  • 1997 — UML 1.0 adopté par l'OMG (Object Management Group) comme standard
  • 2005 — UML 2.0 introduit de nouveaux diagrammes et plus de précision
  • Aujourd'hui — UML 2.5+ est le standard international de modélisation objet
✅ À retenir : UML n'est pas une méthode de développement. C'est un langage de modélisation graphique standardisé, indépendant de toute méthode.

Qu'est-ce qu'UML ?

UML est un langage de modélisation graphique et standardisé qui permet de représenter un système informatique sous différents angles. Il propose 13 types de diagrammes répartis en deux grandes familles :

Diagrammes de structure (statiques)

Ils montrent ce que contient le système :

DiagrammeObjectif
ClassesStructure des classes, attributs, méthodes et relations
ObjetsInstances de classes à un moment donné
ComposantsModules logiciels et leurs dépendances
DéploiementRépartition physique du logiciel (serveurs, nœuds)
PaquetagesOrganisation en packages/namespaces
Structure compositeComposition interne d'une classe
ProfilsExtensions du méta-modèle UML

Diagrammes de comportement (dynamiques)

Ils montrent comment se comporte le système :

DiagrammeObjectif
Cas d'utilisationFonctionnalités vues par les acteurs
ActivitésFlux de travail et processus métier
SéquenceÉchanges de messages dans le temps
CommunicationInteractions entre objets (sans axe temporel)
États-transitionsCycle de vie d'un objet
Vue d'ensemble des interactionsCombinaison de diagrammes d'interaction
⚠️ Important : Dans ce cours, nous étudierons les 4 diagrammes les plus utilisés : Classes, Cas d'utilisation, Activités et Séquence.

Notions clés d'UML

Un diagramme UML c'est quoi ?

  • Une représentation graphique qui s'intéresse à un aspect précis du modèle
  • Chaque type de diagramme possède une structure prédéfinie
  • Il véhicule une sémantique précise (toujours la même vue du système)
  • Combinés, les diagrammes offrent une vue complète (statique + dynamique)

Structure vs Comportement

AspectStructure (statique)Comportement (dynamique)
QuestionDe quoi est composé le système ?Comment fonctionne le système ?
ExemplesClasses, Composants, DéploiementSéquence, Activités, Cas d'utilisation
QuandConception, architectureAnalyse, spécification
💡 Conseil : Commencez toujours par les cas d'utilisation (comprendre les besoins), puis les classes (structurer), et enfin les séquences/activités (décrire les interactions).

Mnémotechnique — Les fondamentaux UML

🧠 « V.C.D.D — Visualiser, Communiquer, Documenter, Détecter »

  • Visualiser → UML transforme les idées abstraites en schémas concrets
  • Communiquer → un langage universel entre développeurs, clients, chefs de projet
  • Documenter → les diagrammes sont la documentation vivante du projet
  • Détecter → trouver les erreurs AVANT de coder (moins cher qu'après !)
