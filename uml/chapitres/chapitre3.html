Module 3

Diagramme de Cas d'utilisation

Apprenez à décrire les fonctionnalités d'un système du point de vue de ses utilisateurs : acteurs, cas d'utilisation, scénarios et relations.

Qu'est-ce qu'un diagramme de cas d'utilisation ?

Le diagramme de cas d'utilisation est comme le menu d'un restaurant : il liste tout ce que le restaurant (système) peut offrir à ses clients (acteurs), sans détailler les recettes (implémentation).

Un diagramme de cas d'utilisation décrit :

  • Les acteurs — entités externes qui interagissent avec le système
  • Les cas d'utilisation — fonctionnalités offertes par le système
  • Le système — le périmètre fonctionnel (rectangle englobant)
  • Des descriptions textuelles détaillées (scénarios)
⚠️ Important : Le diagramme de CU décrit CE QUE fait le système, pas COMMENT il le fait.

Les Acteurs

Un acteur est un élément extérieur au système qui interagit avec lui. C'est un rôle, pas une personne physique.

Caractéristiques des acteurs

  • Un acteur n'est pas forcément humain (peut être un système externe, un dispositif matériel)
  • Une même personne peut jouer plusieurs rôles (Maurice est directeur ET guichetier)
  • Plusieurs personnes peuvent jouer un même rôle (Paul et Pierre sont deux clients)
  • Un acteur exécute un ou plusieurs cas d'utilisation

FormeDescriptionUsage
🧍 Stick manBonhomme bâton + nom dessousLe plus courant
📦 RectangleRectangle avec <<actor>>Systèmes externes

Les Cas d'utilisation

Un cas d'utilisation (Use Case) décrit :

  • Une fonctionnalité du système vue par un acteur
  • Une suite d'interactions entre l'utilisateur et le système
  • Un comportement attendu produisant un résultat observant
  • Un service rendu par le système

Représentation

Un cas d'utilisation est représenté par un ovale contenant son nom. Le nom commence par un verbe à l'infinitif, du point de vue de l'acteur.

Règles importantes

  • Les CU ne doivent pas se chevaucher
  • Un CU doit être relié à au moins un acteur
  • L'ensemble des CU décrit exhaustivement les fonctionnalités du système
✅ Convention : Le nom d'un CU commence par un verbe à l'infinitif : Retirer de l'argent, Consulter un compte, Passer une commande.

Description textuelle et Scénarios

Chaque cas d'utilisation doit être accompagné d'une description textuelle détaillée :

Structure d'une description

RubriqueContenu
TitreNom du cas d'utilisation
RésuméDescription courte de la fonctionnalité
ActeursListe des acteurs impliqués
PréconditionsConditions nécessaires avant le démarrage
Scénario nominalDéroulement normal pas à pas
Scénarios alternatifsVariantes du flux normal
Scénarios d'erreurGestion des cas d'erreur
PostconditionsÉtat du système après exécution

Exemple — Retirer de l'argent

Titre : Retirer de l'argent
Acteur : Client
Précondition : Le distributeur contient des billets, il est en attente

Scénario nominal :
  1. Le client introduit sa carte bancaire
  2. Le système vérifie la validité de la carte
  3. Le système demande le code d'identification
  4. Le client saisit son code
  5. Le système vérifie le code avec la puce
  6. Le système demande le montant du retrait
  7. Le client saisit le montant
  8. Le système vérifie le solde
  9. Le système délivre l'argent et rend la carte

Alternatif (a) : Code erroné (1ère ou 2ème fois)
  → Retour à l'étape 3

Erreur (e) : Carte invalide
  → Carte confisquée, fin du cas d'utilisation

Postcondition : Transaction enregistrée, billets distribués

Relations entre cas d'utilisation

Il existe 3 types de relations entre les cas d'utilisation :

1. Inclusion <<include>>

Le CU source contient obligatoirement le comportement du CU destination.

  • "Retirer de l'argent" include "S'authentifier" → l'authentification est toujours exécutée
  • Permet de factoriser un comportement commun

2. Extension <<extend>>

Le CU source ajoute optionnellement son comportement au CU destination.

  • "Vérifier solde" extend "Effectuer un virement" → la vérification est optionnelle
  • Contrairement à l'inclusion, l'extension est facultative

3. Généralisation

Un CU A est une généralisation d'un CU B si B est un cas particulier de A.

  • "Déposer de l'argent" se spécialise en "Déposer un chèque" et "Déposer du liquide"
  • Le CU parent peut être abstrait (italique)
RelationTypeSymboleCaractère
IncludeFlèche pointillée<<include>>Obligatoire
ExtendFlèche pointillée<<extend>>Optionnel
GénéralisationFlèche triangle vide△Spécialisation
❌ Piège classique : Confondre <<include>> et <<extend>>. Retenez : Include = toujours. Extend = parfois.

TD — Réservation de salles

Dans un établissement scolaire, on gère la réservation des salles et du matériel (ordinateur portable, vidéo-projecteur) :

  • Seuls les enseignants peuvent réserver (sous réserve de disponibilité)
  • Le planning des salles est visible par tout le monde (enseignants + étudiants)
  • Le récapitulatif horaire ne peut être consulté que par les enseignants
  • L'enseignant responsable de formation peut éditer le récapitulatif pour toute la formation
💡 Conseil : Identifiez d'abord les acteurs (qui interagit ?), puis les CU (quelles fonctionnalités ?), et enfin les relations (include, extend, généralisation).
