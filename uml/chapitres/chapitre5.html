Module 5

Diagramme de Séquence

Modélisez les interactions temporelles entre objets : lignes de vie, messages synchrones et asynchrones, fragments d'interaction.

Qu'est-ce qu'un diagramme de séquence ?

Imaginez un échange de SMS entre plusieurs personnes : vous voyez qui envoie quoi à qui, et dans quel ordre. Le diagramme de séquence fait exactement cela, mais pour des objets logiciels.

Le diagramme de séquence représente les interactions entre objets en mettant l'accent sur la dimension temporelle (ordre chronologique des messages).

  • Il montre comment les objets collaborent pour réaliser un scénario
  • Le temps s'écoule de haut en bas
  • Chaque objet a une ligne de vie verticale

Lignes de vie (Lifelines)

Chaque participant à l'interaction est représenté par une ligne de vie :

  • Un rectangle en haut contenant objet : Classe
  • Une ligne verticale pointillée descendant du rectangle
  • Le rectangle d'activation (barre fine) indique quand l'objet est actif

Types de messages

Les messages sont les flèches horizontales entre lignes de vie :

TypeFlècheDescription
Synchrone→ (pleine)L'émetteur attend la réponse avant de continuer (appel de méthode)
Asynchrone⇢ (ouverte)L'émetteur continue sans attendre (signal, événement)
Retour⇠ (pointillé)Réponse à un message synchrone
Création→ vers rectangleCréation d'un nouvel objet (stéréotype ≪create≫)
Destruction✕ sur ligne de vieSuppression de l'objet (croix ✕)
💡 Syntaxe d'un message :
nomMessage(paramètres) : typeRetour
Exemple : getPrice(idProduit) : float

Messages synchrones vs asynchrones

Message synchrone (bloquant)

  • L'émetteur est bloqué en attendant la réponse
  • Représenté par une flèche pleine
  • Correspond à un appel de méthode

Message asynchrone (non bloquant)

  • L'émetteur continue son exécution sans attendre
  • Représenté par une flèche ouverte (non pleine)
  • Correspond à un signal ou événement

Fragments d'interaction

Les fragments combinés permettent de modéliser des comportements complexes :

OpérateurSignificationExemple
altAlternative (if/else)Si montant > solde → refuser, sinon → accepter
optOption (if sans else)Si client fidèle → appliquer remise
loopBoucle (while/for)Pour chaque article du panier → calculer prix
parParallèlePréparer colis ET envoyer facture simultanément
refRéférenceAppeler un sous-diagramme de séquence

Lien avec le diagramme de classes

Le diagramme de séquence et le diagramme de classes sont complémentaires :

  • Les lignes de vie correspondent à des instances de classes
  • Les messages correspondent à des méthodes des classes
  • Si A envoie un message m() à B, la méthode m() doit figurer dans la classe de B
  • Si A communique avec B, il doit exister une association entre les deux classes
✅ Règle d'or : Chaque message du diagramme de séquence doit correspondre à une opération (méthode) existante dans le diagramme de classes. S'il manque, ajoutez-la !

TD — Retrait d'espèces au DAB

Créez un diagramme de séquence pour un retrait d'espèces au distributeur automatique de billets :

  1. Le client insère sa carte dans le DAB
  2. Le DAB demande le code secret au client
  3. Le client saisit son code
  4. Le DAB demande la vérification à la Banque
  5. La Banque vérifie et retourne OK ou KO
  6. Fragment alt :
    • [code OK] → Le DAB demande le montant, le client saisit le montant, le DAB demande l'autorisation à la Banque, la Banque autorise, le DAB distribue les billets
    • [code KO] → Le DAB affiche «code erroné» et restitue la carte

Acteurs/Objets : :Client, :DAB, :Banque

TD — Robot et Bras articulé

Modélisez le diagramme de séquence pour un robot de manipulation :

  • Le Robot commande au Bras de saisir une pièce
  • Le Bras déplace la pince vers la pièce
  • Le Bras ferme la pince (saisie)
  • Le Robot commande le déplacement vers la zone de dépôt
  • Le Bras ouvre la pince (relâche la pièce)
  • Le Bras retourne en position initiale

Identifiez les messages synchrones (Robot attend que le Bras ait terminé) et les messages de retour.

Mnémotechnique — Activité = Flowchart

🧠 « D.A.F.J — Début, Action, Fork/Join, Fin »

  • Début → cercle noir plein ● — un seul point de départ
  • Action → rectangle arrondi — chaque étape du processus
  • Fork / Join → barre horizontale — parallélisme (fork = séparer, join = réunir)
  • Fin → cercle noir dans cercle ⊙ — un ou plusieurs points d'arrivée

Losange ◇ = Décision (comme un carrefour : plusieurs chemins possibles). Les swimlanes séparent les responsabilités par acteur.
