Introduction à UML

// UML = Unified Modeling Language
// Auteurs : Booch (Booch) + Rumbaugh (OMT) + Jacobson (OOSE)
// Standardisé par l'OMG en 1997, v2.0 en 2004

13 diagrammes UML 2.0 :

7 diagrammes de STRUCTURE :
  Classes | Objets | Composants | Déploiement
  Paquetages | Structures composites | Profils

6 diagrammes de COMPORTEMENT :
  Cas d'utilisation | Activités | Machine à états
    └─ Interaction (3) :
       Séquence | Communication | Timing

Diagramme de Classes

Classe UML :
┌──────────────────┐
│ NomClasse         │  ← PascalCase, italique si abstraite
├──────────────────┤
│ - attribut: Type │  ← Attributs
│ + nom: String    │
├──────────────────┤
│ + methode(): void│  ← Méthodes
│ # calculer(): int│
└──────────────────┘

Visibilité :
  +  public       (accessible partout)
  -  private      (classe uniquement)
  #  protected    (classe + sous-classes)
  ~  package      (même paquetage)

Multiplicités :
  1      exactement un       0..1   zéro ou un
  *      zéro ou plusieurs    1..*   un ou plusieurs
  n..m   entre n et m

Relations :
  ────────    Association simple
  ◆────────   Composition  (losange NOIR = vie liée)
  ◇────────   Agrégation   (losange BLANC = vie indépendante)
  ──────▷     Héritage     (triangle vide vers le parent)
  - - - -▷    Réalisation  (implémente une interface)
  - - - ->    Dépendance   (utilisation temporaire)

// Attribut souligné = statique (de classe)
// Nom en italique = abstrait

Cas d'utilisation

Éléments :
  🧍 Acteur       = rôle joué par une entité EXTERNE
  (Ellipse)    = Cas d'utilisation (verbe à l'infinitif)
  [Rectangle]  = Cadre du système (system boundary)
  ─────       = Association acteur ↔ cas

Relations entre CU :
  «include»     ──────>   OBLIGATOIRE (toujours exécuté)
                          Flèche : du CU de base → vers le CU inclus

  «extend»      ──────>   OPTIONNEL (sous condition)
                          Flèche : de l'extension → vers le CU de base

  Généralisation ──▷     Héritage entre CU
                          Flèche : du spécialisé → vers le généralisé

Description textuelle :
  1. Nom du CU          5. Scénario nominal
  2. Acteur principal    6. Scénarios alternatifs
  3. Préconditions       7. Scénarios d'erreur
  4. Déclencheur         8. Postconditions

// Acteur principal = déclenche le CU
// Acteur secondaire = sollicité par le système
// Un acteur peut être : humain, système, timer

Diagramme d'Activités

Symboles :
  ●        Nœud initial (début)
  ▢        Action (rectangle arrondi)
  ◇        Décision / Fusion (losange)
  ━━       Barre de synchronisation (fork / join)
  →        Transition (automatique)
  ⊗        Fin de flux (un seul flux)
  ⊕        Nœud final (TOUS les flux)

Décision :
  ◇ ── [condition vraie] ──→ Action A
  |
  └── [sinon] ──→ Action B
  // Conditions mutuellement exclusives + complètes
  // Toujours prévoir un [sinon]

Fork / Join :
  Fork  = 1 entrée, N sorties  (paralléliser)
  Join  = N entrées, 1 sortie  (synchroniser)
  // Join attend que TOUTES les activités soient terminées

Swimlanes (couloirs) :
  | Acteur A    | Acteur B    |
  | Action 1    |             |
  |       ──────→ Action 2   |
  // Montrent QUI fait QUOI

Diagramme de Séquence

Éléments :
  ┌─────────┐
  │ obj:Cls │   Ligne de vie (lifeline)
  └────┬────┘
       ┆        Ligne pointillée (existence)
       ┃        Rectangle d'activation (exécution)
       ✕        Destruction de l'objet

Messages :
  ──────►  Synchrone   (bloquant, attend réponse)
  ──────>  Asynchrone  (non bloquant, continue)
  <- - - -  Retour      (réponse, pointillé)
  ──────►□  Création    (≪create≫ vers nouveau rectangle)

  Syntaxe : nomMessage(params) : typeRetour

Fragments combinés :
  alt    Alternative (if / else)
  opt    Option (if sans else)
  loop   Boucle (while / for)
  par    Parallèle (exécution simultanée)
  ref    Référence (sous-diagramme)

Règle d'or :
  // Si A envoie m() à B → m() doit être dans la classe de B
  // Si A communique avec B → association dans le diag. de classes
  // Temps s'écoule de HAUT en BAS
